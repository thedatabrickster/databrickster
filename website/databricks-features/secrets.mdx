# Secrets & Secret Scopes

## 1. Executive Summary
Secrets & Secret Scopes provide encrypted, ACL-governed storage for credentials that jobs, notebooks, and pipelines reference at runtime, so passwords, tokens, and keys never live in code or plaintext.

## 2. Primary Purpose
- **Why it exists:** give teams a managed place to store credentials instead of hard-coding them in notebooks, configs, or repos.
- **Business problem:** leaked credentials in source control or shared notebooks are a top cause of breaches and failed audits.
- **Technical problem:** connectors (JDBC, APIs, cloud services) need credentials at runtime without exposing them to code readers, logs, or other users.
- **Scope model:** secrets are grouped into named **scopes**; access is controlled per scope through ACLs (READ / WRITE / MANAGE).

## 3. Where It Fits in the Lakehouse
```text
Developers / Jobs / Notebooks / Pipelines / DLT
        ↓  dbutils.secrets.get(scope, key)
┌───────────────────────────────────────────────┐
│  SECRET SCOPES  (ACL-governed credential store) │
│  scope → key → encrypted value  ·  auto-redact  │
└───────────────────────────────────────────────┘
        ↓                          ↓
Databricks-backed            Key-vault-backed scope
scope (control plane,        (AWS/Azure/GCP KMS or
 encrypted at rest)           Key Vault holds value)
        ↓                          ↓
External systems: DBs · APIs · object storage · SaaS
```
- Sits at the **security edge** between compute and external systems — it hands credentials to code without revealing them.
- A scope is a namespace; the actual value lives either in the Databricks control plane or in an external key vault.
- Referenced at runtime via `dbutils.secrets`, Spark config `{{secrets/scope/key}}` syntax, or the Secrets API/CLI.
- Complements, does not replace, Unity Catalog credentials/connections for governed data access (see section 9).

## 5. Ideal Use Cases
| Scenario | Recommended | Why |
|---|---|---|
| JDBC/ODBC database passwords for jobs | ✅ | Reference at runtime, never in code |
| External API keys (SaaS, payment, LLM providers) | ✅ | Central store with ACL control |
| Cloud storage access keys / SAS tokens | ✅ | Avoids plaintext in cluster config |
| Spark config credentials for mounts | ✅ | `{{secrets/...}}` injection redacts value |
| Service-principal client secrets | ✅ | Rotatable, scoped, not in repos |
| Sharing a credential across a team's jobs | ✅ | One scope, READ ACL to the group |
| Webhook/signing secrets for pipelines | ✅ | Encrypted at rest, auto-redacted |
| Rotating credentials on a schedule | ✅ | Update value, consumers unchanged |
| Backing scope by existing Azure Key Vault | ✅ | Reuse enterprise vault + its rotation |
| Git provider tokens for automation | ✅ | Keeps PAT out of notebooks/config |
| Credentials for Lakeflow Connect sources | ✅ | Managed reference for ingestion |
| Governed table/data-source access (UC) | ⚠️ | Prefer UC connections/credentials instead |
| Storing large config blobs / files | ⚠️ | Value size limited; use storage + a key |
| End-user interactive login secrets | ⚠️ | Not a user password manager |

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Governing access to external data sources | ⚠️ | UC connections/credentials are the governed path |
| Storing certificates or large files | ❌ | Per-secret size limits; not a file store |
| Public/non-sensitive configuration values | ❌ | Use job/cluster params or config tables |
| Cross-account credential brokerage | ⚠️ | Key-vault-backed or UC credentials fit better |
| End-user password vaulting | ❌ | Designed for service credentials, not humans |
| Secrets needing sub-second global rotation | ⚠️ | Rotation is manual/API-driven, not instant everywhere |
| Long-term secret versioning/history | ⚠️ | Databricks-backed scopes keep only current value |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| AWS Secrets Manager | External cloud secret manager | Richer rotation/versioning; needs its own access path and IAM wiring |
| Azure Key Vault | External vault (can back a scope) | Enterprise rotation and RBAC; adds a vault to operate |
| GCP Secret Manager | External cloud secret manager | Native GCP integration; not directly injected by `dbutils` |
| HashiCorp Vault | External secrets platform | Dynamic secrets and leasing; heavier to run and integrate |
| Unity Catalog connections/credentials | In-Databricks governed access | Governs data-source access with UC ACLs; not a general secret store |
| Environment variables | Cluster/config injection | Simple but visible in config and not centrally ACL'd — avoid for secrets |
| Instance profiles / managed identities | Cloud IAM | Credential-free storage access; no secret to store, but cloud-storage-specific |

## 11. Strengths
- Keeps credentials out of code, repos, and shared notebooks.
- Per-scope ACLs (READ / WRITE / MANAGE) enforce least privilege.
- Values encrypted at rest in the control plane.
- Automatic redaction: printed secret values render as `[REDACTED]`.
- Injectable into Spark config with `{{secrets/scope/key}}` syntax.
- Key-vault-backed scopes reuse enterprise vaults and their rotation.
- Native access from `dbutils.secrets`, API, CLI, and SDK.
- Declarable as code via Terraform and Asset Bundles (grants, not values).
- Access and changes captured in Audit Logs.
- No extra licensing cost — part of the platform.

## 12. Limitations
- Databricks-backed scopes store only the current value — no version history.
- Per-secret value size is limited; not for files or certificates.
- ACLs are separate from Unity Catalog grants — a second governance surface to manage.
- Redaction is best-effort; a determined user with READ can still exfiltrate the value in code.
- Rotation is manual or API-driven, not automatic for Databricks-backed scopes.
- Key-vault-backed scopes are read-only from Databricks and cloud-specific in setup.
- Scope names are workspace-scoped; no built-in cross-workspace sharing.
- No native approval workflow or check-out/lease model.
- Deleting a scope is destructive and breaks all consumers immediately.
- MANAGE holders can read and rewrite every secret in the scope.

## 13. Common Mistakes
- Hard-coding credentials in notebooks "temporarily" and forgetting them.
- Granting MANAGE where READ would suffice.
- Assigning ACLs to individual users instead of groups.
- Assuming redaction is absolute security rather than a guardrail.
- Storing non-secret config as secrets, cluttering scopes.
- Using environment variables or cluster env for real credentials.
- Committing secret values into Terraform/Bundle definitions.
- Reusing one scope for unrelated teams and systems.
- Neglecting rotation because "it works today."
- Using secret scopes for governed data access instead of UC connections/credentials.

## 14. Design Recommendations
- ✔ Store every runtime credential in a scope — zero secrets in code.
- ✔ Grant ACLs to groups, not users, and default to READ.
- ✔ Use one scope per system/environment for blast-radius isolation.
- ✔ Back scopes with a cloud key vault when the org already runs one.
- ✔ Declare scopes and ACLs as code; supply values out-of-band.
- ✔ Prefer instance profiles/managed identities for cloud storage access.
- ✔ Prefer Unity Catalog connections/credentials for governed data sources.
- ✔ Establish and automate a rotation schedule via the API.
- ✔ Monitor secret access and scope changes through Audit Logs.

## 18. Cost Considerations
- **Licensing:** included in the platform; no separate charge for secrets. Verify current pricing.
- **Compute:** negligible — secret retrieval is a lightweight control-plane call.
- **Storage:** negligible for Databricks-backed; vault-backed incurs the vault's own storage/API costs.
- **Operational Overhead:** provisioning scopes, managing ACLs, and running rotation are the main ongoing effort.
- **Training:** low — `dbutils.secrets` and CLI are quick to learn; ACL hygiene needs a short standard.

## 19. Executive Takeaway
- ✔ **Best for:** storing service credentials that jobs, notebooks, and pipelines reference at runtime.
- ✔ **Avoid when:** you need governed data-source access (use Unity Catalog) or a full secret-lifecycle platform.
- ✔ **Biggest benefit:** credentials leave the codebase and become ACL-governed and auto-redacted.
- ✔ **Biggest limitation:** no native versioning/auto-rotation and a governance surface separate from Unity Catalog.
- ✔ **Overall recommendation:** adopt as the default for runtime credentials; back with a cloud vault and pair with UC credentials for data access.
