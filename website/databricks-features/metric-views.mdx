# Metric Views

## 1. Executive Summary
Metric Views are Unity Catalog objects that capture reusable, governed definitions of business measures and dimensions in YAML, so every consumer — BI, Genie, and SQL — computes the same numbers from one source of truth.

## 2. Primary Purpose
- **Why it exists:** put the semantic layer inside the lakehouse instead of scattering metric logic across tools.
- **Business problem:** the same KPI (revenue, churn, active users) is defined differently per dashboard, so numbers disagree in meetings.
- **Technical problem:** aggregation, join, and filter logic gets copy-pasted into SQL, BI tools, and prompts, then drifts.
- **Scope:** a governed definition object over existing tables — measures, dimensions, joins — queried like a table but computed consistently.

## 3. Where It Fits in the Lakehouse
```text
Users / BI / Genie / SQL / Notebooks / Apps
        ↓
Compute (Serverless SQL Warehouses, Photon)
        ↓
┌──────────────────────────────────────────────┐
│  UNITY CATALOG                                 │
│   ┌────────────────────────────────────────┐  │
│   │  METRIC VIEW (YAML)                      │ │
│   │  source · joins · dimensions · measures  │ │
│   │  MEASURE() aggregations, semantic layer  │ │
│   └────────────────────────────────────────┘  │
│  grants · lineage · tags · certification       │
└──────────────────────────────────────────────┘
        ↓
Delta Lake tables on object storage
```
- Sits **above governed tables** as a semantic object; consumers query the metric, not raw SQL.
- Aggregations resolve **at query time** through the `MEASURE()` function, so grain is chosen by the consumer.
- Governed by Unity Catalog **grants, lineage, and tags** like any other securable.
- One definition feeds **AI/BI dashboards, Genie spaces, and ad-hoc SQL** identically.

## 5. Ideal Use Cases
| Scenario | Recommended | Why |
|---|---|---|
| One certified definition of core KPIs | ✅ | Measures defined once, consumed everywhere |
| Consistent numbers across BI and Genie | ✅ | Both read the same metric object |
| Grounding natural-language Q&A in trusted logic | ✅ | Genie queries the metric, not ad-hoc SQL |
| Non-additive metrics (ratios, distinct counts) | ✅ | `MEASURE()` recomputes correctly at each grain |
| Star-schema fact + dimension modeling | ✅ | Declarative joins map dimensions to the fact |
| Reusable dimensions across many reports | ✅ | Shared dimension definitions avoid re-derivation |
| Governing metric access with UC grants | ✅ | Metric is a securable with lineage and tags |
| Time-based aggregation at flexible grain | ✅ | Consumers pick day/month/quarter at query time |
| Self-service analytics on curated semantics | ✅ | Business users query safe, named measures |
| Deprecating conflicting dashboard formulas | ✅ | Consolidate to a single definition |
| Certified source for regulatory reporting | ⚠️ | Strong fit; still validate figures and controls |
| Very high-volume sub-second operational lookups | ⚠️ | Query-time aggregation adds cost vs a serving store |
| Complex procedural transformation logic | ⚠️ | Belongs in pipelines, not the semantic layer |
| Cross-platform metric reuse outside Databricks | ⚠️ | Enforcement is UC compute only |

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Heavy ETL / data cleansing logic | ❌ | Use Lakeflow pipelines; metric views are semantic, not transformational |
| Low-latency app serving at scale | ❌ | Aggregations run per query; use a serving table or Lakebase |
| Row/column security enforcement | ❌ | Use ABAC, masks, or dynamic views; metrics are not a security primitive |
| Governing non-Databricks query engines | ❌ | Only UC-aware compute resolves the definition |
| Legacy Hive metastore without UC | ❌ | Metric views are a Unity Catalog object |
| Freeform exploratory one-off analysis | ⚠️ | Ad-hoc SQL is faster than modeling a metric |
| Metrics needing full imperative code | ⚠️ | YAML declarative model has expressiveness limits |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Per-tool metric definitions (in each BI tool) | External | Fast to start, but each tool drifts; no shared source of truth |
| Hand-written SQL / repeated CTEs | In-Databricks | Maximum flexibility, but logic is copied and diverges across queries |
| dbt Semantic Layer / MetricFlow | External | Mature metric framework, but adds a build tool and runs outside UC governance |
| LookML (Looker) | External | Rich BI semantic model, but Looker-only and not queryable by Genie/SQL natively |
| Standard SQL views | In-Databricks | Reusable and governed, but fix grain; no flexible measure aggregation |
| Materialized aggregate tables | In-Databricks | Fast reads, but pre-computed grains and refresh/storage overhead |
| Cube / AtScale / other OLAP layers | External | Cross-platform semantics, but a separate product, cost, and integration |

## 11. Strengths
- Single, governed definition of each metric consumed by BI, Genie, and SQL alike.
- `MEASURE()` recomputes aggregations correctly at any grain the consumer chooses.
- Handles non-additive metrics — ratios, distinct counts — that break naive views.
- Declarative YAML for measures, dimensions, and joins; readable and reviewable.
- Native Unity Catalog securable with grants, lineage, tags, and certification.
- Grounds AI natural-language answers in trusted logic, reducing wrong metrics.
- No data duplication; metrics compute over existing Delta tables.
- Definitions are codifiable in Terraform / Asset Bundles for PR review.
- Star-schema joins map shared dimensions to fact tables once, reused widely.
- Consolidates conflicting dashboard formulas into one endorsed object.

## 12. Limitations
- Aggregation runs at query time, so heavy metrics cost more than pre-materialized tables.
- YAML declarative model is less expressive than full imperative SQL or code.
- Enforcement is in-Databricks; external engines do not honor the definition.
- Requires Unity Catalog and UC-compliant serverless/Photon compute.
- Not a security primitive; row/column controls need ABAC, masks, or dynamic views.
- Newer object than SQL views; some BI and tooling integrations still maturing.
- Modeling correct measures and joins takes upfront analytics-engineering effort.
- Poor fit for low-latency operational serving at high concurrency.
- Underlying table quality is inherited; a bad source yields a bad metric.
- Governance benefit is lost if teams keep parallel definitions in BI tools.

## 13. Common Mistakes
- Treating a metric view as an ETL layer and packing transformation logic into it.
- Defining the same KPI again inside a BI tool, defeating the single source of truth.
- Summing non-additive measures instead of using `MEASURE()` at the right grain.
- Modeling joins that fan out the fact and silently double-count measures.
- Skipping certification and tags, so consumers cannot tell trusted from draft metrics.
- Expecting metric views to enforce row/column security.
- Building on ungoverned or unstable base tables that change under the metric.
- No version control on YAML, so definitions change without review or history.
- Ignoring query cost on very large facts without pre-aggregation strategy.
- Leaving legacy dashboard formulas live in parallel with the new metric.

## 14. Design Recommendations
- ✔ Define each core KPI **once** as a metric view and point all tools at it.
- ✔ Keep metrics **semantic** — push cleansing and shaping into Lakeflow pipelines.
- ✔ Use `MEASURE()` for aggregation; never hardcode a single grain into the metric.
- ✔ Model joins carefully to **avoid fan-out** double counting; validate row counts.
- ✔ **Certify and tag** endorsed metrics so they are discoverable and trusted.
- ✔ Store YAML in **Terraform / Asset Bundles**; review changes via PRs.
- ✔ Connect **Genie spaces and AI/BI dashboards** to the metric, not raw tables.
- ✔ Retire duplicate BI-tool and hand-written definitions after migration.

## 18. Cost Considerations
- **Licensing:** part of Unity Catalog; no separate license. Verify current pricing.
- **Compute:** aggregations resolve at query time on serverless/Photon warehouses (billed as compute); complex measures on large facts add per-query cost.
- **Storage:** none directly; metric views are metadata over existing Delta tables.
- **Operational Overhead:** the real cost is analytics-engineering effort to model measures, dimensions, and joins, then keep them current.
- **Training:** upskilling on the YAML schema, `MEASURE()` semantics, and semantic-layer modeling practices.

## 19. Executive Takeaway
- ✔ **Best for:** establishing one governed, reusable definition of business metrics that BI, Genie, and SQL all compute identically.
- ✔ **Avoid when:** you need heavy transformation, low-latency operational serving, or enforcement outside Databricks.
- ✔ **Biggest benefit:** a single source of truth in Unity Catalog that ends conflicting KPI numbers across tools and AI answers.
- ✔ **Biggest limitation:** query-time aggregation and a declarative YAML model that is less expressive and costlier than pre-materialized tables for heavy metrics.
- ✔ **Overall recommendation:** adopt for core certified KPIs and semantic consistency; model measures carefully, certify and version them, and retire duplicate per-tool definitions.
