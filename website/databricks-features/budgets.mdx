# Budgets & Budget Policies

## 1. Executive Summary
Budgets track account-level Databricks spend against periodic limits and fire threshold alerts, while Budget Policies enforce cost tags on serverless workloads so usage can be attributed and charged back through System Tables.

## 2. Primary Purpose
- **Why it exists:** give finance and platform owners visibility and early warning on Databricks consumption before invoices arrive.
- **Business problem:** cloud data spend is easy to overrun and hard to attribute to teams, projects, or cost centers.
- **Technical problem:** serverless compute has no cluster the user can hand-tag, so usage arrives untagged unless a policy injects the tags.
- **Two distinct features:** Budgets = monitoring + alerts; Budget Policies = tag enforcement/attribution on serverless.

## 3. Where It Fits in the Lakehouse
```text
Users / Jobs / Serverless SQL / Pipelines / Model Serving / Apps
        ↓  (consume DBUs; serverless work carries a budget policy)
Budget Policy  →  injects cost tags on serverless usage
        ↓
system.billing.usage  (metered records + tags, Delta)
        ↓
Budgets  →  aggregate spend vs limit, evaluate thresholds
        ↓
Alerts (email) + BI / chargeback reports over System Tables
```
- Sits in the **administration and monitoring** plane at the account level, above individual workspaces.
- Budget Policies act **upstream** at consumption time; Budgets act **downstream** over metered usage.
- Neither is an enforcement gate on compute — they attribute and warn, they do not cap or kill workloads.
- Chargeback and analysis live in **System Tables** plus BI, not inside the Budgets UI.

## 5. Ideal Use Cases
| Scenario | Recommended | Why |
|---|---|---|
| Early warning on monthly spend overrun | ✅ | Threshold alerts before the invoice |
| Attributing serverless cost to teams | ✅ | Budget Policies inject cost tags |
| Chargeback / showback by cost center | ✅ | Tagged usage in System Tables feeds reports |
| Enforcing a tagging standard on serverless | ✅ | Policy applies tags users cannot skip |
| Per-team or per-project spend tracking | ✅ | Scope budgets by tag or workspace filter |
| Detecting unexpected consumption spikes | ✅ | Period-over-period alerting on usage |
| Finance visibility without cloud console access | ✅ | Account-level view of Databricks spend |
| Governing serverless jobs, SQL, pipelines | ✅ | Policies apply across serverless surfaces |
| Multi-workspace spend rollup | ✅ | Account-level aggregation across workspaces |
| Budgeting a proof-of-concept spend cap awareness | ✅ | Alert when a POC nears its allowance |
| Hard enforcement that stops workloads at a limit | ⚠️ | Budgets alert, they do not block |
| Real-time (sub-minute) cost cutoff | ⚠️ | Usage metering has reporting lag |
| Detailed cost analytics and forecasting | ⚠️ | Use System Tables + BI, not the Budgets UI |
| Attributing classic (non-serverless) compute | ⚠️ | Use cluster/pool custom tags instead of policies |
| Tracking non-Databricks cloud spend | ⚠️ | Out of scope; use cloud billing consoles |

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Hard cap that terminates jobs at a threshold | ❌ | Not an enforcement/kill switch |
| Sub-minute, real-time spend blocking | ❌ | Metering and evaluation are not instant |
| Full cost analytics, trends, forecasting | ❌ | Purpose-built for alerts; use BI over System Tables |
| Tagging classic compute via budget policy | ❌ | Policies target serverless; use cluster tags |
| Governing total cloud (VMs, storage, egress) spend | ❌ | Covers Databricks DBU usage, not raw cloud bills |
| Line-item invoice reconciliation | ⚠️ | Estimates usage; reconcile against provider billing |
| Guaranteed tag coverage on every workload type | ⚠️ | Coverage depends on serverless surface support |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| System Tables + BI dashboards | In-Databricks | Richer analytics and forecasting, but you build alerting yourself. |
| Cluster / pool custom tags | In-Databricks | Tags classic compute, but no serverless coverage or central policy. |
| Cloud billing console (AWS/Azure/GCP) | External | Covers all cloud spend, but lacks DBU-level and per-team Databricks detail. |
| Cloud budget/alert services (e.g. AWS Budgets) | External | Native cloud alerts, but coarse for Databricks attribution. |
| Third-party FinOps platforms | External | Cross-cloud analytics and optimization, but added license and integration. |
| Manual tag governance / spreadsheets | Manual | No license cost, but not enforced, timely, or scalable. |

## 11. Strengths
- Enforces cost tags on serverless usage that users cannot otherwise apply.
- Turns attribution into a governed default rather than a manual convention.
- Account-level rollup across all workspaces in one place.
- Threshold alerts give finance early warning before invoices land.
- Pairs directly with billing System Tables for SQL-based chargeback.
- Low setup effort; configured from the account console or admin APIs.
- Vendor-neutral chargeback model based on standard cost tags.
- Scales with usage without customer-managed infrastructure.
- Separates monitoring (Budgets) from attribution (Policies) cleanly.
- Makes serverless spend as attributable as tagged classic compute.

## 12. Limitations
- Not an enforcement control — alerts do not cap, throttle, or kill workloads.
- Usage metering and budget evaluation carry reporting latency, not real-time.
- Budget Policies apply to serverless surfaces; classic compute uses cluster tags.
- Alerting is threshold email; deep analytics require System Tables + BI.
- Coverage of every serverless surface can vary across releases.
- Amounts are usage estimates; reconcile against provider invoices.
- Only covers Databricks DBU spend, not underlying raw cloud resources.
- Requires account-admin rights to configure, limiting self-service.
- Tag taxonomy must be designed up front or attribution stays messy.
- Historical usage before a policy existed remains untagged.

## 13. Common Mistakes
- Expecting budgets to block spend; they alert, they do not enforce.
- Not assigning budget policies, leaving serverless usage untagged.
- Designing no tag taxonomy, so attribution is inconsistent or unusable.
- Ignoring classic compute tags and assuming policies cover everything.
- Building chargeback in spreadsheets instead of over System Tables.
- Setting a single account budget with no per-team breakdown.
- Treating alert latency as real-time and building cutoffs on it.
- Not restricting who can create or edit policies, weakening the standard.
- Forgetting that pre-policy historical usage cannot be retro-tagged.
- No owner for reviewing alerts, so warnings go unread.

## 14. Design Recommendations
- ✔ Define a cost-tag taxonomy (team, project, environment, cost center) before rollout.
- ✔ Create a default budget policy and require it on all serverless workloads.
- ✔ Set budgets per team or cost center, not just one account-wide limit.
- ✔ Use multiple alert thresholds (for example 50/80/100 percent) for early warning.
- ✔ Build chargeback and trend dashboards over `system.billing.usage`.
- ✔ Restrict policy and budget administration to a small FinOps/platform group.
- ✔ Reconcile budget estimates against cloud provider invoices periodically.
- ✔ Assign a clear owner to triage alerts and review attribution monthly.

## 18. Cost Considerations
- **Licensing:** Budgets and Budget Policies are included with Databricks; no separate license. Verify current pricing.
- **Compute:** the features themselves are not billed; cost comes from running the serverless workloads they track and from queries over usage tables.
- **Storage:** billing System Tables consume minimal managed storage; chargeback marts you build add incremental storage.
- **Operational Overhead:** main effort is designing the tag taxonomy, dashboards, and alert-triage process.
- **Training:** upskill FinOps and platform teams on the billing schema and tag conventions; assign a clear owner.

## 19. Executive Takeaway
- ✔ **Best for:** organizations needing account-level Databricks spend visibility, alerting, and serverless cost attribution for chargeback.
- ✔ **Avoid when:** you need a hard cap that stops workloads or real-time spend blocking — this warns, it does not enforce.
- ✔ **Biggest benefit:** enforced tagging on serverless makes previously opaque usage attributable, then billable, through System Tables.
- ✔ **Biggest limitation:** it is monitoring and attribution only, with reporting latency and no enforcement.
- ✔ **Overall recommendation:** define a tag taxonomy, require a budget policy on all serverless usage, set per-team budgets with tiered alerts, and do the analytics in BI over billing System Tables.
