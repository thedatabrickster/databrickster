# AI Playground

## 1. Executive Summary

AI Playground is a governed, in-workspace UI for interactively testing and comparing foundation models and agents, tuning prompts, exercising tool/function calling, and exporting the equivalent code — a developer-experience tool, not a production serving surface.

## 2. Primary Purpose

- **Why it exists:** give developers a fast, no-setup way to try models and prompts before writing any code.
- **Business problem:** model and prompt selection stalls in ad hoc scripts and vendor consoles, with no shared, governed starting point.
- **Technical problem:** standardizes side-by-side comparison, parameter tuning, and tool-call testing against endpoints already governed by Unity Catalog.
- **Bridge to code:** turns an interactive session into reproducible Python/REST snippets that call Model Serving.

## 3. Where It Fits in the Lakehouse

```text
Developers / AI Engineers
        ↓
Developer Experience   (AI Playground ◀ HERE, Notebooks, Git/Repos)
        ↓
AI & Machine Learning  (Foundation Model APIs, Agent Framework, Vector Search)
        ↓
Serving / Gateway      (Model Serving, AI Gateway, Unity Catalog, Inference Tables)
        ↓
Data Engineering       (Lakeflow, MLflow, Feature Tables)
        ↓
Storage                (Delta Lake, Volumes)
```

- Sits in the **developer-experience** layer — a client that drives serving endpoints, not part of the runtime path.
- Calls **foundation, external, and custom models** already exposed through Model Serving and AI Gateway.
- Every call inherits **Unity Catalog** permissions, quotas, and usage logging — no separate key store.
- Output is **exported code**, so prototypes graduate into notebooks, jobs, or agents.

## 5. Ideal Use Cases

| Scenario | Recommended | Why |
|---|---|---|
| Compare two or more foundation models on the same prompt | ✅ | Side-by-side responses in one view |
| Choose a model for a new feature | ✅ | Quick quality/latency/cost read before committing |
| Iterate on system and user prompts | ✅ | Live editing with immediate output |
| Tune parameters (temperature, max tokens, top-p) | ✅ | Interactive sliders, no code round-trip |
| Test tool / function calling behavior | ✅ | Exercise tool schemas and inspect call flow |
| Prototype a retrieval or agent flow | ✅ | Attach tools and Vector Search before building |
| Export a working prompt to Python or REST | ✅ | Generates code against Model Serving |
| Onboard developers to available models | ✅ | Discover governed endpoints in one place |
| Sanity-check an external LLM (OpenAI, Anthropic) | ✅ | Routed through governed external-model endpoints |
| Reproduce and triage a bad model response | ✅ | Replay prompt/params to isolate the cause |
| Demo model behavior to stakeholders | ✅ | No local setup or keys required |
| Draft prompts for a Genie or BI use case | ⚠️ | Useful for wording; validate in that product |
| Bulk / batch evaluation across a dataset | ⚠️ | Use MLflow evaluation or `ai_query` instead |
| Regression testing a prompt over time | ⚠️ | Manual; move to automated eval harness |

## 6. Poor Fit

| Scenario | Avoid? | Reason |
|---|---|---|
| Production or user-facing inference | ❌ | Interactive UI, not a served endpoint |
| Programmatic / high-volume calls | ❌ | Use Model Serving REST or `ai_query` |
| Automated CI/CD prompt regression | ❌ | No scripting or scheduling surface |
| Large-scale offline batch scoring | ❌ | Not built for throughput |
| Model training or fine-tuning | ❌ | Serving-side only; no training loop |
| Systematic quantitative eval at scale | ⚠️ | MLflow evaluation is the right tool |
| Long-lived multi-user shared sessions | ⚠️ | Session state is ephemeral, single-user |
| Storing prompts as the system of record | ⚠️ | Persist in Git/Repos, not the Playground |

| Depends on / Works with | Role (one sentence) |

## 10. Architecture Alternatives

| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Databricks Notebooks + SDK | In-Databricks | Full flexibility, but manual setup and no side-by-side UI. |
| MLflow evaluation / LLM-as-judge | In-Databricks | Quantitative, repeatable eval vs interactive exploration. |
| OpenAI Playground | External | Fast for OpenAI models, but outside UC governance and lineage. |
| Vendor consoles (Anthropic, Google, Azure OpenAI) | External | Per-vendor UI; no unified access or exported Databricks code. |
| Local notebooks / scripts | External | Total control, but scattered keys and no shared governance. |
| Hosted prompt tools (e.g. LangSmith) | External / OSS | Rich prompt ops, but separate from lakehouse data and access. |

## 11. Strengths

- Zero setup — compare models and prompts in the browser, no code or keys.
- Side-by-side model comparison on identical inputs in one view.
- Interactive parameter tuning without code round-trips.
- Tool / function-calling testing against real tool schemas.
- Calls inherit Unity Catalog permissions, quotas, and logging automatically.
- One place to reach foundation, external, and custom endpoints.
- Export to Python/REST shortens the prototype-to-code path.
- Lowers the barrier for developers new to available models.
- Governed access to external LLMs without scattering vendor keys.
- Useful for reproducing and triaging problematic model responses.

## 12. Limitations

- Not a production surface — no SLA, scheduling, or programmatic access.
- Single-user, ephemeral sessions; state is not durably shared.
- No built-in bulk or dataset-level evaluation.
- Manual and qualitative — not a substitute for automated eval.
- Limited to endpoints exposed through Model Serving.
- No native prompt versioning; rely on Git/Repos externally.
- Comparison depth is bounded by the UI, not scriptable.
- Cost visibility per experiment is coarse without gateway reporting.
- Advanced agent orchestration still needs the Agent Framework.
- Feature and model availability vary by region and workspace tier.

## 13. Common Mistakes

- Treating exported snippets as production code without hardening.
- Using it for batch or high-volume calls instead of `ai_query`.
- Skipping Unity Catalog setup, so few or no endpoints appear.
- Comparing models on one prompt and over-generalizing the result.
- Not saving good prompts into Git/Repos before the session ends.
- Confusing interactive tool tests with a fully built agent.
- Relying on it for quantitative eval instead of MLflow.
- Ignoring gateway quotas and hitting throttling mid-session.
- Assuming external-model access without configured endpoints.
- Sharing sensitive data in prompts without checking guardrails.

## 14. Design Recommendations

- ✔ Expose the models developers actually need via Model Serving first.
- ✔ Front all endpoints with AI Gateway for quotas, guardrails, and logging.
- ✔ Register external LLMs as external-model endpoints, never raw keys.
- ✔ Use the Playground to shortlist, then confirm with MLflow evaluation.
- ✔ Export prototypes to notebooks and commit prompts to Git/Repos.
- ✔ Test tool schemas here before wiring them into an agent.
- ✔ Set and communicate per-user quotas to control experimentation cost.
- ✔ Treat it as a starting point, not the system of record for prompts.

## 18. Cost Considerations

- **Licensing:** no separate license; part of the workspace. Verify current pricing.
- **Compute:** each call consumes model inference — per-token for foundation/external models, DBUs for provisioned endpoints; interactive use is low-volume but not free.
- **Storage:** negligible; sessions are ephemeral. Inference Tables, if enabled, add Delta storage.
- **Operational Overhead:** minimal — no infra to run; effort is endpoint setup and gateway quota policy.
- **Training:** low — developers self-serve; teams already on Model Serving/Unity Catalog adopt quickly. External-model routes add per-provider token charges.

## 19. Executive Takeaway

- ✔ **Best for:** interactive prototyping and comparison of models, prompts, and tool calls before writing production code.
- ✔ **Avoid when:** you need production inference, batch scoring, or automated evaluation.
- ✔ **Biggest benefit:** zero-setup, governed exploration that exports straight into Databricks code.
- ✔ **Biggest limitation:** developer-experience only — no SLA, scale, or programmatic surface.
- ✔ **Overall recommendation:** adopt as the default starting point for model and prompt selection; hand off to Model Serving for runtime and MLflow for formal evaluation.
