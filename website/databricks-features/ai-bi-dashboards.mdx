# AI/BI Dashboards

## 1. Executive Summary
AI/BI Dashboards are Databricks-native dashboards that build visualizations directly on Unity Catalog-governed data, execute on SQL warehouses, and pair with Genie for natural-language analysis — no separate BI server required.

## 2. Primary Purpose
- **Why it exists:** provide first-party dashboarding on the Lakehouse so BI runs on the same governed data as engineering and ML.
- **Business problem solved:** removes the cost, extract copies, and split governance of standing up a separate BI platform for internal reporting.
- **Technical problem solved:** interactive charts, filters, and sharing served straight from Delta via SQL warehouses, with access enforced centrally.

## 3. Where It Fits in the Lakehouse
```text
Users / Executives / BI Analysts / Business Users
        ↓
AI/BI Dashboards  +  Genie (natural language)
        ↓
Databricks SQL  →  SQL Warehouses (Serverless / Pro / Classic) + Photon
        ↓
Unity Catalog   (permissions · lineage · masking · audit)
        ↓
Delta Lake      (tables · materialized views · metric views)
```
- AI/BI Dashboards are the **native visualization/consumption layer** — a read front door for business users.
- They render on SQL warehouse compute; they add no separate BI server or engine.
- Genie sits alongside as the natural-language companion over the same governed data.
- Datasets and permissions inherit from Unity Catalog — dashboards are not a new governance island.

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Existing large Power BI/Tableau estate | ⚠️ | Keep incumbent; use DBSQL as the source |
| Highly customized/branded external portals | ⚠️ | Embedding and theming are more constrained |
| Complex multi-source blended reporting | ⚠️ | Best on Lakehouse data; blend upstream first |
| Operational OLTP dashboards (ms refresh) | ❌ | Warehouse is analytical, not sub-second OLTP |
| Advanced statistical/custom-viz workbench | ❌ | Use notebooks or a specialized viz tool |
| Offline/disconnected report consumption | ❌ | Serving requires the platform online |
| Rich write-back / planning applications | ❌ | Dashboards are read-oriented, not write-back |

## 11. Strengths
- Dashboards run directly on Unity Catalog-governed Delta data — no extract copies.
- No separate BI server to license, deploy, or operate.
- SQL warehouse backing gives Photon performance and elastic concurrency.
- Access, masking, and audit inherit from Unity Catalog automatically.
- Pairs natively with Genie for natural-language Q&A on the same data.
- Native sharing and secure embedding for governed consumers.
- Metric and materialized views enable governed, fast KPIs.
- Fast to stand up for teams already on Databricks.
- Schedules and subscriptions distribute snapshots without extra tooling.
- Ideal for operational/monitoring dashboards over system tables.

## 12. Limitations
- Modeling and layout are lighter than Power BI, Tableau, or Looker.
- Best on Lakehouse data; multi-source blending should happen upstream.
- Embedding and theming are more constrained than dedicated BI portals.
- No offline consumption; serving requires the platform online.
- Read-oriented — no rich write-back or planning workflows.
- Cost and latency track SQL warehouse sizing and idle settings.
- Advanced/custom visualization types are limited versus specialist tools.
- External/anonymous distribution is less flexible than incumbent BI.
- Feature availability can vary by cloud and region.
- Migrating from a mature BI estate is a real project, not a toggle.

## 13. Common Mistakes
- Recreating a mature Power BI/Tableau estate wholesale instead of targeting new/internal reporting.
- Querying raw tables per widget rather than using metric or materialized views.
- Oversizing warehouses instead of enabling multi-cluster autoscaling.
- Setting aggressive idle timeouts that discard caching benefits.
- Granting broad table access rather than governed views, row filters, and masks.
- Treating dashboards as OLTP for sub-second operational refresh.
- Ignoring Genie as the better path for ad hoc natural-language questions.
- Blending many external sources at the dashboard layer instead of upstream.
- Leaving legacy DBSQL dashboards unmigrated and fragmenting reporting.

## 14. Design Recommendations
- ✔ Use AI/BI Dashboards for internal reporting on Lakehouse data; keep incumbent BI for its existing estate.
- ✔ Back dashboards with **serverless** warehouses sized small with autoscaling.
- ✔ Model KPIs once in metric views and reuse across dashboards and Genie.
- ✔ Precompute hot results with materialized views to cut cost and latency.
- ✔ Enforce access through Unity Catalog views, row filters, and column masks.
- ✔ Point natural-language users to Genie on the shared warehouse.
- ✔ Build cost/usage monitoring dashboards on system tables.
- ✔ Migrate legacy DBSQL dashboards to consolidate on one surface.

## 18. Cost Considerations
- **Licensing:** no separate BI-server license; consumption-based DBUs on the backing warehouse. Verify current pricing.
- **Compute:** dominant cost; controlled via warehouse size, autoscaling, and idle timeout. Serverless bills only while active.
- **Storage:** Delta on your cloud object store plus any materialized-view footprint; independent of compute.
- **Operational Overhead:** low — no BI server to deploy or patch; governance managed in Unity Catalog.
- **Training:** low for SQL/BI teams; standard SQL plus Unity Catalog and dashboard concepts.

## 19. Executive Takeaway
- ✔ **Best for:** internal, governed reporting and self-service directly on Lakehouse data, paired with Genie.
- ✔ **Avoid when:** you need a mature multi-source BI estate, pixel-perfect layouts, write-back, or OLTP refresh.
- ✔ **Biggest benefit:** dashboards on UC-governed data with no separate BI server or extract copies.
- ✔ **Biggest limitation:** lighter modeling, layout, and external-distribution options than Power BI, Tableau, or Looker.
- ✔ **Overall recommendation:** adopt AI/BI Dashboards as the default native BI layer for Databricks-resident data, keeping incumbent BI tools for their established estates.
