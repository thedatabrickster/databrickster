# Mosaic AI Model Training

## 1. Executive Summary

Mosaic AI Model Training is a managed service for fine-tuning and continued pre-training of foundation and open-source LLMs on your governed data, producing a Unity Catalog model ready for Model Serving.

## 2. Primary Purpose

- **Why it exists:** adapt a base LLM to a domain, task, or style using in-house data without building a training stack.
- **Business problem:** off-the-shelf models miss company terminology, tone, and formats; API fine-tuning ships proprietary data to third parties.
- **Technical problem:** self-managed GPU training (data prep, distributed orchestration, checkpointing, evaluation) is slow and error-prone.
- **Governance angle:** training data, run lineage, and the output model stay inside Unity Catalog under existing permissions.

## 3. Where It Fits in the Lakehouse

```text
Users / Data Scientists / AI Engineers
        ↓
Developer Experience   (Notebooks, SDK, MLflow API)
        ↓
AI & Machine Learning  (Mosaic AI: Model Training ◀ HERE → Model Serving, Agents)
        ↓
Governance             (Unity Catalog: registered model, data, lineage)
        ↓
Data Engineering       (Lakeflow, Feature tables, MLflow tracking)
        ↓
Storage                (Delta Lake, Volumes)
```

- Sits in the **model-production** part of the AI/ML layer — the step before serving, not the serving edge.
- Reads training data from **Delta tables or Volumes** and writes the resulting model to **Unity Catalog** via MLflow.
- Consumes governed data in place; it does not expose endpoints — that is **Model Serving**.
- Runs on Databricks-managed GPU capacity; no cluster fleet to size or maintain.

## 5. Ideal Use Cases

| Scenario | Recommended | Why |
|---|---|---|
| Domain adaptation (legal, clinical, finance jargon) | ✅ | Teaches vocabulary and patterns prompting cannot reliably inject |
| Enforcing a fixed output format or schema | ✅ | Fine-tuning bakes structure in more consistently than prompts |
| Consistent brand voice / tone at scale | ✅ | Style transfers into weights, shrinking prompt overhead |
| Classification / extraction with labeled data | ✅ | Small tuned model can beat a large prompted one on cost and accuracy |
| Shrinking a large prompted model to a smaller one | ✅ | Distill behavior into a cheaper base to cut inference cost |
| Keeping proprietary training data in-house | ✅ | Training runs inside the governance boundary, not a vendor API |
| Continued pre-training on a large private corpus | ✅ | Instruction tuning alone cannot absorb broad new knowledge |
| Reducing latency/tokens by trimming few-shot prompts | ✅ | Learned behavior removes long in-context examples |
| Teaching a niche task with thousands of examples | ✅ | Enough labeled data to move base-model behavior |
| Multilingual or code-dialect specialization | ✅ | Targeted data shifts model competence in a domain |
| Rapidly changing facts / fresh knowledge | ⚠️ | RAG updates instantly; retraining lags and costs more |
| Very small dataset (few hundred examples) | ⚠️ | May overfit; try prompting or RAG first |
| Need transparent, per-answer source citations | ⚠️ | Retrieval gives provenance; weights do not |
| One-off or exploratory task | ⚠️ | Prompt engineering is faster and cheaper to iterate |

## 6. Poor Fit

| Scenario | Avoid? | Reason |
|---|---|---|
| Injecting frequently updated knowledge | ❌ | Use RAG / Vector Search; retraining cannot keep pace |
| Requirement for citable, auditable sources | ❌ | Fine-tuned weights offer no provenance trail |
| Tiny or low-quality labeled dataset | ❌ | Insufficient signal; overfitting and regression risk |
| Fixing a single bad prompt output | ❌ | Prompt/few-shot iteration is faster and reversible |
| Real-time personalization per user | ❌ | Retraining cadence is far too slow; use context/retrieval |
| No ML skills or evaluation discipline on team | ⚠️ | Tuning without eval harms quality silently |
| Base model already meets quality with prompting | ⚠️ | Added cost and lifecycle with little marginal gain |
| Highly volatile requirements | ⚠️ | Each change implies a new run and re-eval cycle |

## 10. Architecture Alternatives

| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| RAG / Vector Search | In-Databricks | Fresh, citable knowledge with no training; weaker on style/format/behavior. |
| Prompt / few-shot engineering | In-Databricks | Fastest and cheapest to iterate; costs tokens and caps consistency. |
| Foundation Model APIs (as-is) | In-Databricks | Zero training effort; no domain adaptation. |
| OpenAI / Together / Fireworks fine-tuning | External | Managed and simple; proprietary data leaves the governance boundary. |
| Self-managed GPU training (PyTorch, DeepSpeed) | External / OSS | Maximum control; you own orchestration, scaling, and reliability. |
| Cloud vendor tuning (SageMaker, Vertex) | External | Deep cloud integration; model and lineage live outside Unity Catalog. |

## 11. Strengths

- Managed distributed training — no GPU cluster, orchestration, or checkpoint plumbing to run.
- Training data and output model stay inside the Unity Catalog governance boundary.
- Supports full fine-tuning, instruction tuning, and continued pre-training on open models.
- Direct MLflow-to-Unity-Catalog-to-Model-Serving path shortens time to production.
- Run lineage links dataset, base model, and resulting version for audit.
- Output is a portable, self-owned model — no per-call dependence on a tuning vendor.
- Smaller tuned models can cut inference cost versus a large prompted base.
- Integrates with Agent Evaluation to gate promotion on measured quality.
- Retraining pipelines can be scheduled through Lakeflow Jobs.
- Data never leaves for a third-party fine-tuning API.

## 12. Limitations

- Requires ML skill: dataset design, hyperparameters, and evaluation discipline.
- Fine-tuning teaches behavior, not fresh facts — knowledge goes stale between runs.
- No per-answer source citations, unlike retrieval-based approaches.
- GPU training cost and run time are significant versus prompting or RAG.
- Supported base models and regions are constrained to the managed catalog.
- Overfitting and capability regression are real risks with small or skewed data.
- Each requirement change implies a new run, re-eval, and redeploy cycle.
- Results are hard to explain; weight changes are opaque.
- Iteration loop is slower than editing a prompt or retrieval index.
- Tightly coupled to the Databricks platform for the training path.

## 13. Common Mistakes

- Fine-tuning to inject knowledge that RAG would handle better and fresher.
- Training on a dataset too small or noisy to move behavior reliably.
- Skipping a held-out evaluation set, so quality regressions go unnoticed.
- No baseline comparison against the prompted base model before committing.
- Confusing style/format goals (good fit) with factual recall (poor fit).
- Ignoring data quality and label consistency — garbage in, garbage out.
- Underestimating retraining cadence and its recurring compute cost.
- Bypassing Unity Catalog registration, breaking lineage and access control.
- Not versioning datasets, making runs irreproducible.
- Treating a tuned model as static and never re-evaluating against drift.

## 14. Design Recommendations

- ✔ Exhaust prompting and RAG first; fine-tune only for behavior they cannot deliver.
- ✔ Curate and version training data as Delta tables under Unity Catalog.
- ✔ Hold out a labeled evaluation set and gate promotion on measured gains.
- ✔ Always benchmark the tuned model against the prompted base model.
- ✔ Register every run and model in Unity Catalog via MLflow for lineage.
- ✔ Start with the smallest base that can meet quality to control cost.
- ✔ Deploy tuned models through Model Serving, not ad-hoc hosting.
- ✔ Automate periodic retraining with Lakeflow Jobs when data shifts.
- ✔ Pair fine-tuning with RAG when both behavior and fresh facts are needed.

## 18. Cost Considerations

- **Licensing:** consumption-based via DBUs; no separate license. Verify current pricing.
- **Compute:** GPU training billed by run duration and model size; the dominant, spiky cost. Amortize over inference savings from a smaller tuned model.
- **Storage:** training datasets, checkpoints, and model versions consume Delta/Volume storage — set retention.
- **Operational Overhead:** medium — managed training removes infra work, but data curation, evaluation, and retraining cadence are ongoing.
- **Training (skills):** requires data scientists / AI engineers fluent in dataset design and evaluation; budget for that capability, not just compute.

## 19. Executive Takeaway

- ✔ **Best for:** adapting an open or foundation LLM's style, format, or domain behavior on governed in-house data.
- ✔ **Avoid when:** the need is fresh, citable knowledge, a quick prompt fix, or you lack labeled data and evaluation discipline.
- ✔ **Biggest benefit:** a self-owned, governed model produced without a training stack and without shipping data to a tuning vendor.
- ✔ **Biggest limitation:** teaches behavior, not facts — knowledge goes stale, and each change means a new run, re-eval, and redeploy.
- ✔ **Overall recommendation:** reach for it after prompting and RAG fall short on consistency, format, or cost; pair with RAG when both behavior and fresh facts matter.
