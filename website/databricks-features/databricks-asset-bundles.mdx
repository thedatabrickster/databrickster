# Databricks Asset Bundles (DABs)

## 1. Executive Summary
Databricks Asset Bundles (DABs) are the native infrastructure-as-code format that packages Databricks resources — jobs, pipelines, model endpoints, apps, and their code — into a versioned `databricks.yml` project deployed to per-environment targets (dev/staging/prod) via the Databricks CLI and Git-based CI/CD.

## 2. Primary Purpose
- **Why it exists:** replace click-ops and ad-hoc REST scripts with a declarative, reviewable definition of Databricks resources.
- **Business problem:** manually built jobs/pipelines drift between environments, aren't auditable, and can't be reliably promoted dev → prod.
- **Technical problem:** package source code plus resource config together, parameterize per environment, and deploy atomically and repeatably from CI.
- **Positioning:** the Databricks-recommended IaC path for application-level resources; complements (not replaces) Terraform for account/workspace provisioning.

## 3. Where It Fits in the Lakehouse
```text
Developer / Git Repo
        ↓
databricks.yml (bundle definition + targets)
        ↓
Databricks CLI  (validate → deploy → run)
        ↓
Workspace Resources → Lakeflow Jobs · Pipelines · Model Serving · Apps · Volumes
        ↓
Unity Catalog (governance, lineage)
        ↓
Delta Lake
```
- Sits in the **deployment / developer-experience layer**, above the resources it provisions and below the Git/CI system that drives it.
- Declares resources but does not run workloads itself — it deploys definitions the platform executes.
- Deployed artifacts (code, config) land in the workspace file system and are governed by Unity Catalog.

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
| --- | --- | --- |
| Terraform Databricks provider | External IaC | Broader account/infra scope; more verbose, no app-code packaging or bundled build |
| REST API / SDK scripts | Custom scripting | Maximum control; you build idempotency, state, and promotion yourself |
| Manual UI (click-ops) | Interactive | Fast to start; no versioning, review, or reproducible promotion |
| dbt projects | Modeling framework | Owns SQL model deploys; does not manage jobs, endpoints, or apps |
| dbx (legacy) | Predecessor tool | Superseded by DABs; no longer recommended for new work |
| CI-invoked notebooks | Imperative | Simple but drifts; no declarative diff or teardown |

## 11. Strengths
- Declarative single source of truth for Databricks resources in Git.
- Per-target parameterization (variables, overrides) for clean dev/prod separation.
- Native, first-party — tracks new resource types as the platform evolves.
- `validate` catches config errors before deploy; `deploy` is repeatable and idempotent.
- Packages application code (wheels, notebooks) alongside resource config.
- Fits any CI/CD runner through the CLI; no proprietary orchestrator required.
- Per-user dev targets isolate developer deployments automatically.
- Templates (`bundle init`) standardize project structure across teams.
- Clean teardown via `bundle destroy` for ephemeral environments.
- Service-principal deployment supports non-interactive, auditable prod releases.

## 12. Limitations
- Scope is Databricks resources — not workspaces, metastores, or cloud networking.
- No rich state/plan model like Terraform; drift detection is limited.
- Unity Catalog grant/governance coverage is narrower than the Terraform provider.
- YAML complexity grows with variables, targets, and includes on large projects.
- Debugging deploy failures can mean reading CLI output and workspace state.
- Newer resource types may lag full bundle support at release.
- Requires CLI version discipline; behavior can shift across CLI releases.
- No native secret management — relies on secret scopes and external stores.
- Rollback is redeploy-a-prior-commit, not an automatic snapshot revert.
- Learning curve for teams new to IaC and Git-based workflows.

## 13. Common Mistakes
- Hardcoding environment values instead of using targets and variables.
- Deploying prod with a personal user identity rather than a service principal.
- Committing secrets into `databricks.yml` instead of secret scopes.
- Skipping `bundle validate` in CI, letting bad config reach the workspace.
- Editing deployed jobs in the UI, causing drift from the bundle.
- Overusing DABs for account-level infra that belongs in Terraform.
- One giant bundle for the whole org instead of scoped, team-owned bundles.
- Pinning no CLI version, so CI behavior changes silently across runs.

## 14. Design Recommendations
- ✔ Keep one bundle per deployable unit or team domain, not one per org.
- ✔ Use `dev`, `staging`, `prod` targets with variables for all environment differences.
- ✔ Deploy prod via a service principal in CI, never an interactive user.
- ✔ Run `validate` on every pull request; `deploy` only on protected branches.
- ✔ Pin the Databricks CLI version in CI for reproducible deploys.
- ✔ Store secrets in secret scopes and reference them, never inline.
- ✔ Reserve Terraform for workspace/metastore/network; DABs for resources.
- ✔ Use `bundle init` templates to enforce structure across new projects.

## 18. Cost Considerations
- **Licensing:** DABs and the Databricks CLI carry no separate license fee; verify current pricing.
- **Compute:** cost comes from the deployed resources (jobs, pipelines, endpoints), not from bundles themselves.
- **Storage:** minimal — deployed artifacts and config in the workspace file system.
- **Operational Overhead:** authoring and maintaining YAML, CI pipelines, and CLI version management.
- **Training:** upfront investment for teams new to IaC and Git-based deployment workflows.

## 19. Executive Takeaway
- ✔ **Best for:** teams promoting jobs, pipelines, and endpoints dev → prod through Git-based CI/CD.
- ✔ **Avoid when:** provisioning workspaces or account-level infra — use the Terraform provider instead.
- ✔ **Biggest benefit:** one declarative, version-controlled source of truth that ends click-ops drift.
- ✔ **Biggest limitation:** scope stops at Databricks resources with limited state and governance coverage.
- ✔ **Overall recommendation:** adopt as the default deployment path for Databricks application resources; pair with Terraform for infrastructure.
