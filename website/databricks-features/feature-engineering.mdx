# Feature Store / Feature Engineering in Unity Catalog

## 1. Executive Summary
Feature Engineering in Unity Catalog turns Delta tables into governed, discoverable, reusable feature stores that join automatically into training and inference, eliminating train/serve skew and carrying lineage from raw data through features to deployed models.

## 2. Primary Purpose
- **Why it exists:** to make ML features a shared, governed asset rather than logic re-implemented per model and per team.
- **Business problem solved:** cuts duplicated feature work, speeds new-model delivery, and provides audit-ready lineage from data to prediction.
- **Technical problem solved:** guarantees the same feature computation at training and serving time, removing a leading cause of silent model degradation.

## 3. Where It Fits in the Lakehouse
```text
Raw / Bronze Delta
        ↓
Feature pipelines (batch/streaming, Lakeflow)
        ↓
Feature tables in Unity Catalog  (Delta, PK-keyed)
        ↓         ↘
Training set join        Online Store / online table
   ↓                          ↓
MLflow model (logged w/ feature refs)
        ↓
Model Serving  →  auto feature lookup at inference
```
- Sits between curated Delta data and the ML model as the **feature governance and reuse layer**.
- One feature table feeds both offline training joins and online low-latency lookups.
- Feature references are logged with the model, so serving resolves lookups automatically.

| Features reused across multiple models | ✅ | Single governed source, no duplication |
| Preventing train/serve skew | ✅ | Same computation offline and online |
| Real-time scoring needing fresh lookups | ✅ | Online store serves sub-millisecond reads |
| Point-in-time correct training joins | ✅ | Time-series lookups avoid label leakage |
| Model lineage and audit requirements | ✅ | Feature-to-model lineage in Unity Catalog |
| Cross-team feature discovery | ✅ | Catalog search and documentation |

| Scenario | Avoid? | Reason |
| --- | --- | --- |

| Raw unstructured blob storage | ❌ | Store computed features, not raw media |
| Single throwaway experiment | ⚠️ | Setup cost may not pay back |

| Teams unwilling to own pipelines | ⚠️ | Stale tables erode trust fast |
| Extreme ultra-low-latency edge scoring | ⚠️ | Confirm online store meets the SLA |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
| --- | --- | --- |
| Ad-hoc feature code in notebooks | In-Databricks | Fast to start, but duplicates logic and invites skew |

## 11. Strengths
- Same feature computation at training and serving removes train/serve skew.
- Feature-to-model lineage is captured natively in Unity Catalog.
- One feature table serves both offline training and online inference.
- Point-in-time joins prevent label leakage in time-series training.
- Governed access: table- and column-level grants, audit, discovery.
- Serving auto-resolves logged feature references, less glue code.
- Batch and streaming pipelines keep features fresh.
- Reuse across teams cuts duplicated feature engineering.
- Feature Serving exposes lookups to apps beyond model endpoints.
- Built on Delta and Spark, so it scales with existing compute.

## 12. Limitations
- Online store adds infrastructure, sync latency, and cost to manage.
- Point-in-time correctness requires disciplined timestamp keys.
- Teams must own pipeline freshness; stale tables silently hurt models.
- Not a general OLTP or app database, narrow to feature workloads.
- Schema and key changes to shared tables ripple to many consumers.
- Online store latency/throughput limits need validation per SLA.
- Best experience assumes Unity Catalog is fully adopted.
- Migrating legacy Workspace Feature Store tables takes planning.
- Unstructured inputs must be pre-converted to computed features.
- Cross-cloud/external serving may need extra sync patterns.

## 13. Common Mistakes
- Recomputing features separately for serving, reintroducing skew.
- Skipping point-in-time joins and leaking future data into training.
- No ownership for pipeline refresh, letting features go stale.
- Storing raw data instead of governed, model-ready features.
- Overloading one table with unrelated features and poor keys.
- Ignoring online store cost and TTL until bills or latency spike.
- Not logging feature references, forcing manual lookup at serving.
- Treating the feature store as an afterthought instead of a design layer.
- Uncontrolled schema changes breaking downstream models.

## 14. Design Recommendations
- ✔ Define features once as governed tables and reuse across models.
- ✔ Use point-in-time / time-series lookups for temporal training data.
- ✔ Log feature references with the model so serving auto-joins them.
- ✔ Assign explicit owners and SLAs for each feature pipeline.
- ✔ Provision online stores only for models needing real-time lookup.
- ✔ Version and document feature definitions in Unity Catalog.
- ✔ Choose primary keys and timestamp keys deliberately up front.
- ✔ Monitor freshness, drift, and online store latency/cost.
- ✔ Refresh features via Lakeflow pipelines and Jobs, not manual runs.

## 18. Cost Considerations
- **Licensing:** included with Databricks ML capabilities; no separate feature store license. Verify current pricing.
- **Compute:** feature pipeline runs (batch/streaming) and training joins are the main drivers.
- **Storage:** offline tables are Delta; online stores add duplicate hot storage for served features.
- **Operational Overhead:** owning pipeline freshness, keys, and online sync is the real ongoing cost.
- **Training:** moderate learning curve for point-in-time joins, keys, and online store setup.

## 19. Executive Takeaway
- ✔ **Best for:** organizations reusing ML features across teams and models while needing consistent training and serving with governed lineage.
- ✔ **Avoid when:** you have a single throwaway model, analyst-only SQL work, or need a general-purpose application database.
- ✔ **Biggest benefit:** the same governed features power training and serving, eliminating train/serve skew with lineage to every model.
- ✔ **Biggest limitation:** it requires disciplined pipeline ownership and, for real-time use, an online store that adds cost and operational work.
- ✔ **Overall recommendation:** adopt as the standard feature layer once Unity Catalog is in place and features are shared beyond a single model.
