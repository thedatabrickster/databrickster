# Compliance Security Profile

## 1. Executive Summary
The Compliance Security Profile is a workspace/account setting that enables a bundle of hardened security controls and enhanced security monitoring so a Databricks environment can meet regulated standards such as FedRAMP Moderate/High, HIPAA, PCI-DSS, and IRAP.

## 2. Primary Purpose
- **Why it exists:** provide a one-switch, provider-attested baseline that satisfies external auditors instead of hand-assembling controls.
- **Business problem:** regulated workloads (health, payments, government) cannot run without demonstrable, standard-aligned hardening.
- **Technical problem:** ad-hoc hardening drifts, is hard to prove, and misses controls (FIPS crypto, image hardening, host monitoring).
- **Scope:** applied per compliance standard at the workspace level, backed by account-level settings and enforced through control-plane policy.

## 3. Where It Fits in the Lakehouse
```text
Users / Admins / Auditors / Service Principals
        ↓  (governed access)
Account Console (settings + compliance standards)
        ↓
┌───────────────────────────────────────────────┐
│  COMPLIANCE SECURITY PROFILE                    │
│  hardened images · FIPS crypto · restricted     │
│  controls · enhanced host + workload monitoring │
└───────────────────────────────────────────────┘
        ↓                          ↓
Compute (clusters / SQL /       Enhanced Security
serverless) run hardened        Monitoring agents
        ↓                          ↓
Unity Catalog + Delta (governed data, at rest/in transit)
```
- Sits at the **security/administration** layer — it constrains and instruments the whole environment, it does not process data.
- Acts as a **policy envelope**: once on, compute launches with hardened images and stricter defaults.
- Pairs enforcement (restricted controls) with evidence (enhanced monitoring, audit trail).
- Applies beneath governance (Unity Catalog) and above raw compute and storage.

| HIPAA-regulated PHI processing | ✅ | Hardening plus monitoring supports safeguards |
| PCI-DSS cardholder data environments | ✅ | Restricted controls and FIPS crypto align to requirements |
| Government / public-sector data platforms | ✅ | Meets IRAP/FedRAMP-style expectations |
| Handling classified or high-sensitivity data | ✅ | Hardened images reduce host attack surface |
| Audit-driven security posture standardization | ✅ | One consistent, provable baseline across workspaces |
| Multi-workspace regulated estate | ✅ | Applied per workspace with account-level consistency |
| Requiring FIPS 140-2/3 validated encryption | ✅ | Enforces validated cryptographic modules |
| Enhanced runtime threat monitoring | ✅ | Adds host/workload security agents |
| Contractual security-standard obligations | ✅ | Demonstrates named-standard adherence |
| General enterprise hardening (no named regime) | ⚠️ | Works, but adds constraints you may not need |
| Latency-sensitive workloads | ⚠️ | Monitoring and hardening add minor overhead |
| Environments needing bleeding-edge features | ⚠️ | Some newer features/regions may be unsupported |
| Cost-sensitive dev/test sandboxes | ⚠️ | Constraints and overhead rarely justified there |
| Non-regulated analytics teams | ⚠️ | Restrictions can slow iteration without benefit |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Manual workspace hardening | In-Databricks/manual | Full control, but drifts and is hard to attest. |
| Cluster Policies + custom controls | In-Databricks | Constrains compute, but not a named-standard baseline. |
| Cloud compliance tooling (Config, Defender, SCC) | External | Broad cloud coverage, but lacks Databricks-internal hardening. |
| Separate isolated/dedicated accounts | External/architecture | Strong blast-radius isolation, but higher cost and duplication. |
| Third-party CSPM / posture tools | External | Added detection and reporting, but another integration. |
| GovCloud / sovereign region deployment | External/architecture | Regulatory boundary, but region and feature limits. |

## 11. Strengths
- One setting enables a provider-attested, standard-aligned control bundle.
- Hardened base images reduce host and OS attack surface.
- Enforces FIPS-validated cryptography for regulated encryption needs.
- Adds enhanced security monitoring beyond standard audit logging.
- Restricts risky configuration so environments stay within policy.
- Consistent baseline reproducible across many workspaces.
- Reduces manual hardening effort and configuration drift.
- Backed by Databricks compliance attestations auditors recognize.
- Applies per standard, so scope matches each workload's regime.
- Integrates with Unity Catalog, CMK, and private networking.

## 12. Limitations
- Not all features or runtimes are supported under every profile.
- Region availability is limited, especially for FedRAMP High and GovCloud.
- Restricted controls can block workflows teams previously relied on.
- Enhanced monitoring and hardening add compute and cost overhead.
- Covers platform controls only — not your policies, processes, or people.
- Enabling or changing standards can require compute restarts or recreation.
- Some serverless and preview capabilities may be unavailable.
- Higher-tier plans (often Enterprise) are typically required.
- Troubleshooting is harder inside a locked-down, monitored environment.
- Does not by itself deliver certification; auditors still assess the whole system.

## 13. Common Mistakes
- Treating the profile as full compliance rather than one control layer.
- Enabling it in dev/test where it adds cost without benefit.
- Not checking feature and region support before committing workloads.
- Assuming it replaces Unity Catalog data-access governance.
- Skipping customer-managed keys and private networking that standards expect.
- Ignoring the added monitoring cost and operational overhead.
- Applying the wrong standard for the actual regulatory obligation.
- Forgetting that restricted controls may break existing automation.
- No plan for the compute restart/recreation that enablement triggers.
- Failing to align workspace scoping with the data's sensitivity.

## 14. Design Recommendations
- ✔ Map each workload to its actual regime before selecting a standard.
- ✔ Enable the profile at initial workspace setup, not after go-live.
- ✔ Pair it with Unity Catalog, CMK, private networking, and IP access lists.
- ✔ Verify feature and region support for your required capabilities first.
- ✔ Isolate regulated workloads in dedicated, profile-enabled workspaces.
- ✔ Feed enhanced monitoring and audit logs into your SIEM.
- ✔ Budget for the added monitoring and hardening overhead.
- ✔ Document the profile as evidence within your control matrix.

## 18. Cost Considerations
- **Licensing:** typically requires a higher tier (often Enterprise) and specific standard support; verify current pricing.
- **Compute:** hardened images and monitoring agents add modest per-node overhead versus standard compute.
- **Storage:** minimal direct impact; encryption and CMK usage may add small key-management cost.
- **Operational Overhead:** feature/region constraints and stricter controls increase design, testing, and support effort.
- **Training:** security and platform teams must learn the profile's constraints, evidence, and troubleshooting patterns; assign a clear owner.

## 19. Executive Takeaway
- ✔ **Best for:** regulated workloads (FedRAMP, HIPAA, PCI-DSS) needing an attested, hardened, monitored baseline.
- ✔ **Avoid when:** the environment has no compliance obligation and the constraints only add cost and friction.
- ✔ **Biggest benefit:** one setting delivers standard-aligned hardening plus enhanced monitoring across the environment.
- ✔ **Biggest limitation:** feature and region support is constrained and it does not by itself grant certification.
- ✔ **Overall recommendation:** enable it at setup for regulated workspaces, pair it with Unity Catalog, CMK, and private networking, and treat it as one documented layer of a broader compliance program.
