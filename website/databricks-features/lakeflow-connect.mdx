# Lakeflow Connect

## 1. Executive Summary

Lakeflow Connect is a set of managed, low-code connectors that ingest data from SaaS applications and operational databases into governed Delta tables, using change data capture to keep them incrementally in sync.

## 2. Primary Purpose

- **Why it exists:** eliminate hand-built ingestion pipelines for common enterprise sources.
- **Business problem solved:** slow, costly, and fragile onboarding of source systems into the Lakehouse.
- **Technical problem solved:** incremental extraction, CDC, schema drift, and connector maintenance for SaaS and databases.
- **Family note:** part of **Lakeflow** alongside **Lakeflow Declarative Pipelines** (transform) and **Lakeflow Jobs** (orchestration); Connect handles the ingest stage.

## 3. Where It Fits in the Lakehouse

```text
Sources → Lakeflow Connect → Delta (Bronze) → Lakeflow Declarative Pipelines → Delta (Silver/Gold) → SQL / BI / AI
 (Salesforce, Workday,     (managed CDC       (raw landed)     (transform + DQ)      (curated)          (consumers)
  SQL Server, ...)          ingestion)
```

- Sits at the **ingestion** edge of the Data Engineering layer.
- Lands source data as governed Delta tables registered in Unity Catalog.
- Feeds downstream transformation (Declarative Pipelines) and consumption (SQL, BI, AI).
- Uses CDC where the source supports it, so only changed rows move after the initial snapshot.

## 5. Ideal Use Cases

| Scenario | Recommended | Why |
|---|---|---|
| Salesforce CRM into the Lakehouse | ✅ | Managed SaaS connector |
| Workday HR/finance ingestion | ✅ | Managed SaaS connector |
| SQL Server operational DB replication | ✅ | Database connector with CDC |
| Incremental sync after initial snapshot | ✅ | CDC moves only changed rows |
| Landing Bronze tables for medallion ETL | ✅ | Outputs governed Delta |
| Replacing per-source custom extractors | ✅ | Reduces bespoke code |
| Schema-drift-tolerant ingestion | ✅ | Managed schema evolution |
| Governed, Unity Catalog-registered sources | ✅ | Native UC integration |
| Consolidating third-party ETL spend | ✅ | Fewer external tools |
| Low-code teams without pipeline engineers | ✅ | UI/API-driven setup |
| Serverless, hands-off ingestion ops | ✅ | Managed compute |
| Source not on the supported connector list | ⚠️ | Use Auto Loader/JDBC instead |
| Sub-second operational replication | ⚠️ | Scheduled, not event-instant |
| Heavy in-flight transformation needs | ⚠️ | Transform downstream, not here |
| One-off file loads | ⚠️ | Auto Loader/COPY INTO is simpler |

## 6. Poor Fit

| Scenario | Avoid? | Reason |
|---|---|---|
| Unsupported / niche source systems | ❌ | Connector may not exist |
| Complex transformation during ingest | ❌ | Connect lands raw; transform later |
| Sub-second replication SLAs | ⚠️ | Scheduled incremental, not streaming |
| Ad-hoc file ingestion from storage | ⚠️ | Auto Loader / COPY INTO fits better |
| Full control over extraction logic | ⚠️ | Managed model abstracts internals |
| Off-Databricks portability requirement | ❌ | Runtime-bound to the platform |
| Reverse ETL (warehouse to SaaS) | ❌ | Inbound ingestion only |

## 10. Architecture Alternatives

| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Auto Loader | In-Databricks | Great for files/object storage; not SaaS/DB CDC. |
| JDBC + custom Spark ingestion | In-Databricks | Full control; you own CDC, drift, and maintenance. |
| COPY INTO | In-Databricks | Simple batch file loads; no managed connectors or CDC. |
| Fivetran / Airbyte | External | Broad connector catalog; extra tool, cost, and governance seam. |
| Debezium + Kafka | External | Low-latency log-based CDC; heavy infra to operate. |
| Cloud-native ETL (ADF / Glue) | External | Managed pipelines; weaker Lakehouse-native lineage. |

## 11. Strengths

- Managed connectors remove per-source extraction and CDC code.
- Incremental CDC after initial snapshot reduces data movement and cost.
- Native Unity Catalog governance and lineage on ingested tables.
- Low-code setup via UI and API lowers the engineering barrier.
- Managed schema evolution handles common source drift.
- Serverless execution removes cluster sizing and ops for ingestion.
- Consolidates ingestion into the platform, cutting external tool sprawl.
- Direct fit as the Bronze-landing stage of the medallion pattern.
- Integrates with Lakeflow Jobs for end-to-end orchestration.
- Deployable as code via Databricks Asset Bundles.

## 12. Limitations

- Connector catalog is finite — unsupported sources need Auto Loader/JDBC.
- Scheduled incremental model, not sub-second operational replication.
- Lands raw data; transformation must happen downstream.
- Runtime-bound to Databricks — no off-platform portability.
- Managed abstraction limits fine-grained control of extraction.
- Source-side CDC prerequisites (log access, permissions) can be involved.
- Connector coverage and features still maturing versus older ETL vendors.
- Inbound only — no reverse ETL back to source systems.
- Network setup for on-prem databases adds initial complexity.
- Per-source quotas and object limits may apply.

## 13. Common Mistakes

- Expecting connectors for every niche source system.
- Treating it as a transformation tool instead of pure ingestion.
- Skipping source-side CDC prerequisites, then hitting sync failures.
- Assuming sub-second latency from a scheduled incremental model.
- Not landing to Bronze and transforming with Declarative Pipelines.
- Hand-managing credentials instead of using secrets and PrivateLink.
- Ignoring schema-drift handling settings on volatile sources.
- Not orchestrating ingestion with downstream jobs, causing stale data.

## 14. Design Recommendations

- ✔ Use Connect for supported SaaS/DB sources; Auto Loader for files.
- ✔ Land raw to Bronze, then transform with Lakeflow Declarative Pipelines.
- ✔ Verify source CDC prerequisites before onboarding a database.
- ✔ Secure connectivity with PrivateLink/gateways and store secrets centrally.
- ✔ Register targets in Unity Catalog for governance and lineage.
- ✔ Orchestrate ingestion and transforms together via Lakeflow Jobs.
- ✔ Deploy connector definitions with Databricks Asset Bundles.
- ✔ Fall back to JDBC/custom ingestion only where no connector exists.

## 18. Cost Considerations

- **Licensing:** consumed as Databricks DBUs; connector/serverless tier affects rate — verify current pricing.
- **Compute:** serverless ingestion compute; CDC incremental sync keeps steady-state cost lower than full reloads.
- **Storage:** standard Delta storage for landed tables plus CDC history overhead.
- **Operational Overhead:** low — managed connectors, schema evolution, and retries reduce ops effort.
- **Training:** low to moderate — teams learn connector setup and source prerequisites.

## 19. Executive Takeaway

- ✔ **Best for:** managed, low-code ingestion of supported SaaS apps and databases into the Lakehouse with CDC.
- ✔ **Avoid when:** the source is unsupported, you need sub-second replication, or you require off-platform portability.
- ✔ **Biggest benefit:** removes custom ingestion and CDC code while keeping data governed in Unity Catalog.
- ✔ **Biggest limitation:** finite connector catalog and scheduled incremental (not real-time) model.
- ✔ **Overall recommendation:** default ingestion path for supported sources on Databricks; pair with Declarative Pipelines and Jobs. See the [feature catalog](/databricks-features) for related components.
