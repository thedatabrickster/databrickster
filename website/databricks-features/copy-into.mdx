# COPY INTO

## 1. Executive Summary

COPY INTO is an idempotent SQL command that incrementally and retryably loads files from cloud storage into an existing Delta table, skipping files it has already ingested.

## 2. Primary Purpose

- **Why it exists:** give SQL-first teams a simple, re-runnable way to load files into Delta without streaming or custom code.
- **Business problem solved:** repeated or accidental re-runs that duplicate data, and the effort of hand-tracking which files were loaded.
- **Technical problem solved:** per-file dedup, format/schema options, and load retries expressed in one declarative statement.
- **Scope note:** it is a *load command*, not a pipeline — it appends into a target you already created.

## 3. Where It Fits in the Lakehouse

```text
Cloud Storage (S3/ADLS/GCS) → COPY INTO (SQL) → Delta (Bronze) → Silver/Gold → SQL / BI / AI
   (landing files)             (idempotent load)    (raw ingest)    (curation)    (consumers)
```

- Sits at the **ingestion edge** of Data Engineering — a batch hop from files to Delta.
- Typically appends to the **Bronze** (raw) table; downstream transforms build Silver/Gold.
- Runs in Databricks SQL, notebooks, or Lakeflow Jobs on any warehouse or cluster.
- Governs the target through Unity Catalog once data lands in Delta.

## 5. Ideal Use Cases

| Scenario | Recommended | Why |
|---|---|---|
| One-time bulk load of a static dataset | ✅ | Single re-runnable statement |
| Periodic/scheduled batch file loads | ✅ | Idempotent per file on each run |
| SQL-first teams without Spark expertise | ✅ | Pure SQL, no streaming code |
| Idempotent, retryable ingestion | ✅ | Skips already-loaded files automatically |
| Loading a known, bounded set of files | ✅ | Explicit source path and pattern |
| Modest file counts per load | ✅ | Directory listing is efficient here |
| Well-defined CSV/JSON/Parquet formats | ✅ | Format options set inline |
| Backfilling a new Delta table | ✅ | Loads existing files once, safely |
| Ad hoc reloads during development | ✅ | Re-run without duplicating rows |
| Load step inside a Lakeflow Job | ✅ | Schedulable SQL task |
| Light schema evolution on ingest | ✅ | Supports mergeSchema and evolution options |
| Continuous, low-latency file arrival | ⚠️ | Auto Loader fits streaming better |
| Millions of files per directory | ⚠️ | Listing degrades; use Auto Loader notifications |
| Database or SaaS source ingestion | ⚠️ | Use Lakeflow Connect, not a file loader |
| Force-reloading unchanged files | ⚠️ | Requires explicit copy options |

## 6. Poor Fit

| Scenario | Avoid? | Reason |
|---|---|---|
| Continuous streaming ingestion | ⚠️ | Batch command; Auto Loader is event-driven |
| Very high file counts (millions) | ❌ | Directory listing does not scale like notifications |
| Database / CDC / API sources | ❌ | Not a file loader; use Lakeflow Connect |
| In-place edits to loaded files | ❌ | Tracks file identity, not content changes |
| Complex multi-table transforms | ❌ | It loads; transformation belongs downstream |
| Sub-second latency requirements | ❌ | Runs per invocation, not per event |
| Row-level updates/deletes to target | ⚠️ | Append-oriented; use MERGE separately |

## 10. Architecture Alternatives

| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Auto Loader | In-Databricks (streaming) | Scales to millions of files and continuous arrival; more setup and streaming concepts. |
| Lakeflow Connect | In-Databricks | Managed DB/SaaS connectors; not a raw file loader. |
| INSERT INTO ... SELECT | In-Databricks (SQL) | Full control per run; you own dedup and file tracking. |
| CREATE TABLE AS SELECT | In-Databricks (SQL) | Good for one-shot creation; not incremental or idempotent across runs. |
| Spark batch read + write | In-Databricks | Flexible transforms; re-reads everything, no built-in load state. |
| Azure Data Factory / AWS Glue | External | Managed orchestration; less Lakehouse-native, added tooling and cost. |
| Fivetran / Airbyte | External | Broad source coverage; separate platform and per-file bookkeeping. |

## 11. Strengths

- Idempotent by default — re-running skips already-loaded files.
- Pure SQL; no Spark or streaming knowledge required.
- Retryable: failed loads can be safely re-run without duplication.
- Single statement replaces custom file-tracking logic.
- Supports CSV, JSON, Parquet, ORC, Avro, text, and binary formats.
- Inline format and copy options tune parsing per load.
- Optional schema evolution and mergeSchema on ingest.
- Runs on SQL warehouses, notebooks, or as a scheduled job task.
- Governed source and target access through Unity Catalog.
- Low learning curve and fast to stand up for periodic loads.

## 12. Limitations

- Batch-oriented — not built for continuous, low-latency streaming.
- Directory listing degrades at very high file counts.
- Target Delta table must exist before loading.
- Append-focused; updates/deletes need a separate MERGE.
- Tracks file identity, not in-place content changes.
- Not a file source for databases, CDC, or APIs.
- Transformation is limited to the load SELECT; modeling belongs downstream.
- Force-reload of unchanged files requires explicit copy options.
- Load history metadata grows on the target over time.
- Less throughput at scale than Auto Loader notification mode.

## 13. Common Mistakes

- Reaching for COPY INTO at scale where Auto Loader notifications fit.
- Assuming it creates the target table; it must exist first.
- Setting force-reload options, then unintentionally duplicating data.
- Expecting in-place file edits to be re-ingested.
- Over-transforming in the load statement instead of writing raw Bronze.
- Using it for database/SaaS sources instead of Lakeflow Connect.
- Ignoring schema evolution options, then breaking on new columns.
- Pointing at ungoverned paths without an external location or volume.
- Loading many tiny files without downstream compaction.

## 14. Design Recommendations

- ✔ Create the target Delta table first, then append with COPY INTO.
- ✔ Use it for one-time, periodic, or SQL-first batch loads.
- ✔ Rely on default idempotency; avoid force-reload unless intentional.
- ✔ Enable schema evolution/mergeSchema for drifting file formats.
- ✔ Write raw to Bronze; defer cleansing and modeling to Silver/Gold.
- ✔ Access sources through Unity Catalog volumes or external locations.
- ✔ Schedule recurring loads as a Lakeflow Jobs SQL task.
- ✔ Switch to Auto Loader for continuous or very high-volume ingestion.

## 18. Cost Considerations

- **Licensing:** consumed as Databricks DBUs via the SQL warehouse or cluster running the command — verify current pricing.
- **Compute:** batch/triggered compute only; no idle streaming cost between loads.
- **Storage:** standard Delta target storage plus load-history metadata on the table.
- **Operational Overhead:** low — one statement, no checkpoints or notification queues to manage.
- **Training:** minimal — teams already fluent in SQL adopt it quickly.

## 19. Executive Takeaway

- ✔ **Best for:** simple, idempotent, retryable batch loading of files into Delta for SQL-first teams and periodic loads.
- ✔ **Avoid when:** you need continuous streaming, very high file volumes, or database/SaaS ingestion (use Auto Loader or Lakeflow Connect).
- ✔ **Biggest benefit:** one re-runnable SQL command replaces custom file-tracking and prevents duplicate loads.
- ✔ **Biggest limitation:** batch and append-oriented — it does not scale like Auto Loader or handle updates and streaming.
- ✔ **Overall recommendation:** default file-load command for smaller, periodic, SQL-driven Bronze ingestion. See the [feature catalog](/databricks-features) for related components.
