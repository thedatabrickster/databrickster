# Instance Pools

## 1. Executive Summary
Instance Pools are managed sets of ready-to-use, idle cloud VMs that Databricks clusters draw from at start-up, cutting launch latency and cloud-acquisition variability in exchange for paying cloud (not DBU) cost on the idle instances they hold.

## 2. Primary Purpose
- **Why it exists:** eliminate the cloud VM acquisition delay that dominates cold cluster start time.
- **Business problem:** slow, unpredictable cluster starts waste analyst/engineer time and stall scheduled jobs.
- **Technical problem:** cloud provider capacity is not instant — VMs must be requested, booted, and joined to a cluster on every cold start.
- **Scope:** provides warm VMs for all-purpose clusters, job clusters, and (indirectly) SQL warehouses; does not run workloads itself.

## 3. Where It Fits in the Lakehouse
```text
Users / Jobs / Schedulers
        ↓  (start / scale cluster)
Cluster Manager (control plane)
        ↓  (request nodes)
┌───────────────────────────────────────────────┐
│  INSTANCE POOL  (warm, idle VM reservoir)       │
│  min idle · max capacity · instance type · tags │
└───────────────────────────────────────────────┘
        ↓  (attach pre-booted VM in seconds)
Cluster (driver + workers) → Spark / Photon
        ↓
Unity Catalog → Delta / cloud storage
```
- Sits **below the cluster, above the cloud provider** — a buffer that holds VMs so clusters skip cold acquisition.
- Purely a compute-provisioning optimization; it has no role in data access, governance, or query execution.
- Idle instances in the pool incur cloud infrastructure cost but **no DBUs** until a cluster claims them.
- One pool typically backs many clusters sharing the same instance type and runtime.

## 5. Ideal Use Cases
| Scenario | Recommended | Why |
|---|---|---|
| High-frequency scheduled jobs needing fast starts | ✅ | Warm VMs cut per-run start latency |
| Many short-lived job clusters throughout the day | ✅ | Amortizes idle cost across frequent reuse |
| Interactive teams sensitive to cluster start time | ✅ | Sub-minute starts improve iteration |
| Bursty workloads with predictable peak windows | ✅ | Pre-warm before peaks, shrink idle after |
| Standardizing instance types across many clusters | ✅ | Pool fixes type, driving consistency |
| Reducing exposure to cloud capacity errors | ✅ | Pre-acquired VMs buffer transient shortages |
| Spot/preemptible workloads needing on-demand fallback | ✅ | Pool can mix spot with on-demand backfill |
| CI/CD pipelines spinning ephemeral clusters | ⚠️ | Helps only if runs are frequent enough |
| Fleet with uniform node-type requirements | ✅ | One pool serves many identical clusters |
| Cost-attribution via consistent pool tags | ✅ | Pool tags propagate to borrowed VMs |
| Latency-critical SLA jobs | ✅ | Removes acquisition time from the SLA path |
| Large one-off batch backfill | ⚠️ | Benefit limited to a single start |
| Rarely used ad-hoc analysis cluster | ❌ | Idle cost outweighs infrequent starts |
| Workloads already on serverless compute | ❌ | Serverless has no user-managed pool to warm |
| Highly variable instance-type needs | ⚠️ | Multiple pools add management overhead |

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Serverless SQL / jobs / notebooks | ❌ | Databricks manages warm capacity; no pool needed |
| Low-frequency, unpredictable usage | ❌ | Idle VMs sit unused, pure cost with no return |
| Single instance type never reused | ⚠️ | Little reuse to amortize idle cost |
| Teams unwilling to monitor utilization | ⚠️ | Untuned pools quietly waste spend |
| Highly diverse GPU/CPU/memory mixes | ⚠️ | Each type needs its own pool to be effective |
| Environments optimizing purely for lowest cost | ❌ | Any idle capacity conflicts with the goal |

## 9. Architecture Dependencies
| Depends on / Works with | Role (one sentence) |
|---|---|
| Clusters / Compute | The consumers that borrow VMs from the pool on start-up. |
| Lakeflow Jobs | Schedules job clusters that attach to a pool for fast starts. |
| Cluster Policies | Constrain and mandate pool usage, instance types, and idle limits. |
| Cloud Provider (AWS/Azure/GCP) | Supplies the underlying VM capacity the pool reserves. |
| Databricks Runtime | Preloaded runtime versions speed the attach step further. |
| System Tables / Billing | Report idle-instance cost and pool utilization for tuning. |
| Tags | Propagate to borrowed instances for cost attribution. |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Serverless compute | In-Databricks | No pool to manage and near-instant starts, but less config control and different pricing. |
| On-demand clusters (no pool) | In-Databricks | Zero idle cost, but full cold-start latency each launch. |
| Cloud reserved instances / savings plans | Cloud provider | Lower per-hour cost via commitment, but no launch-latency benefit. |
| Cluster reuse / long-running clusters | In-Databricks | Avoids restarts entirely, but pays for full compute while idle, not just VMs. |
| Preloaded Docker / custom images | In-Databricks | Speeds environment setup, but does not remove VM acquisition time. |
| Cloud provider warm pools | Cloud provider | Similar warming outside Databricks, but no cluster-attach integration. |

## 11. Strengths
- Cuts cluster start time from minutes to seconds when a warm VM is available.
- Idle instances cost cloud infrastructure only — **no DBUs** accrue until borrowed.
- Buffers against transient cloud capacity/quota errors.
- Shared across many clusters, amortizing idle cost over frequent reuse.
- Preloadable runtime versions further shorten attach time.
- Standardizes instance type and configuration across a fleet.
- Simple to enable and integrate with existing job and interactive clusters.
- Tags propagate to borrowed VMs for consistent cost attribution.
- Supports spot/preemptible instances with on-demand fallback.
- Enforceable through Cluster Policies for consistent adoption.

## 12. Limitations
- Idle instances cost money even when no cluster borrows them.
- Benefit disappears when the pool is empty (cold miss) — sizing must anticipate demand.
- One pool serves a single instance type; diverse fleets need many pools.
- No value for serverless workloads, where Databricks manages capacity.
- Requires ongoing monitoring to keep idle cost and hit rate balanced.
- Relevance is shrinking as serverless compute expands across the platform.
- Does not reduce the DBU or compute cost of the workload itself.
- Min-idle instances hold cloud capacity that could count against quotas.
- Preloaded runtime versions consume storage on pooled VMs.
- Adds a configuration surface that can drift from policy over time.

## 13. Common Mistakes
- Setting min-idle too high, paying for VMs that rarely get borrowed.
- Setting min-idle to zero, so every start is still a cold miss.
- Creating a pool per team when one shared pool would suffice.
- Forgetting to set a max capacity, letting the pool grow without bound.
- Ignoring idle-instance cost in billing until the invoice arrives.
- Mismatching pool instance type to actual workload needs.
- Applying pools to serverless-eligible workloads where they add no value.
- Not preloading the runtime version clusters actually use.
- Leaving idle-instance auto-termination windows too long.
- Never revisiting sizing as workload patterns change.

## 14. Design Recommendations
- ✔ Size min-idle to your typical concurrent cold-start demand, not peak.
- ✔ Set a hard max capacity to cap idle-cost exposure.
- ✔ Consolidate to a few shared pools per common instance type.
- ✔ Preload the runtime versions clusters actually use.
- ✔ Set a short idle auto-termination window for above-min instances.
- ✔ Track hit rate and idle cost via System Tables; retune monthly.
- ✔ Enforce pool usage through Cluster Policies.
- ✔ Reassess pools whenever serverless becomes viable for the workload.

## 18. Cost Considerations
- **Licensing:** no separate license; pools are a native compute feature. Verify current pricing.
- **Compute:** idle pooled VMs incur cloud infrastructure cost but no DBUs; DBUs apply only once a cluster borrows and runs.
- **Storage:** preloaded runtime versions and disk on pooled VMs consume cloud storage.
- **Operational Overhead:** ongoing tuning of min-idle, max capacity, and idle timeout to balance latency vs cost.
- **Training:** low — familiar to anyone who manages Databricks clusters and policies.

## 19. Executive Takeaway
- ✔ **Best for:** frequent, latency-sensitive cluster starts sharing a common instance type.
- ✔ **Avoid when:** usage is rare or the workload can run on serverless compute.
- ✔ **Biggest benefit:** near-instant cluster starts by removing cloud VM acquisition time.
- ✔ **Biggest limitation:** idle instances cost money whether or not they are borrowed.
- ✔ **Overall recommendation:** a proven cost-vs-latency lever for busy classic-compute fleets, but weigh it against serverless, whose growth is steadily narrowing where pools add value.
