# Unity Catalog

## 1. Executive Summary
Unity Catalog is the account-level governance layer that provides a single permission, lineage, discovery, and audit model for all data and AI assets across every Databricks workspace.

## 2. Primary Purpose
- **Why it exists:** replace per-workspace, per-tool access controls with one consistent governance model.
- **Business problem:** fragmented ownership, unclear lineage, and audit gaps that block compliance (GDPR, HIPAA, SOX) and slow data sharing.
- **Technical problem:** identities, grants, and metadata previously lived in disconnected Hive metastores, table ACLs, and cloud IAM — impossible to reason about centrally.
- **Scope:** governs tables, views, volumes (files), ML models, functions, and now AI assets (agents, vector indexes) under one namespace.

## 3. Where It Fits in the Lakehouse
```text
Users / BI / Notebooks / AI Agents / SQL / Apps
        ↓
Compute (Serverless, Clusters, SQL Warehouses)
        ↓
┌─────────────────────────────────────────────┐
│  UNITY CATALOG  (account-level control plane) │
│  identity · grants · lineage · audit · ABAC   │
│  three-level namespace: catalog.schema.object │
└─────────────────────────────────────────────┘
        ↓
External Locations + Storage Credentials
        ↓
Delta Lake / Parquet / Volumes on cloud object storage
```
- Sits **between compute and storage** — every read/write is authorized and logged here.
- **Account-scoped, not workspace-scoped:** one metastore serves many workspaces per region.
- Owns the mapping from logical names to physical cloud storage via storage credentials + external locations.
- It is the enforcement point, not a query engine — compute (Photon/Spark/DBSQL) does the work under UC's grants.

## 5. Ideal Use Cases
| Scenario | Recommended | Why |
|---|---|---|
| Single governance model across many workspaces | ✅ | Account-level metastore unifies grants |
| Fine-grained access (row/column) on tables | ✅ | Row filters + column masks native |
| Attribute-based access at scale | ✅ | ABAC policies via governed tags |
| End-to-end column-level lineage | ✅ | Captured automatically at runtime |
| Regulatory audit (who accessed what) | ✅ | System tables + audit logs |
| Governing ML models and features | ✅ | Models/features are UC objects |
| Governing GenAI agents & vector indexes | ✅ | AI assets live in the namespace |
| Cross-org data sharing | ✅ | Delta Sharing built on UC |
| Governed file/unstructured data | ✅ | Volumes provide managed paths |
| Querying external sources in place | ✅ | Lakehouse Federation registers them |
| Self-service data discovery/search | ✅ | Catalog Explorer + metadata tags |
| Central business metric definitions | ✅ | Metric Views are UC objects |
| Multi-cloud single namespace | ⚠️ | Metastore is per-region/per-cloud |
| Sub-millisecond OLTP authorization | ⚠️ | Optimized for analytics, not OLTP path |
| Governing assets fully outside Databricks | ⚠️ | Federation helps but coverage varies |

## 6. Poor Fit
| Scenario | Avoid? | Reason |
|---|---|---|
| Use as a general-purpose IAM for non-data apps | ❌ | Scoped to data/AI assets, not app identity |
| Replacement for cloud KMS/secret storage | ❌ | Use Secrets + cloud KMS instead |
| Single global namespace across all regions | ⚠️ | One metastore per region |
| Legacy Hive metastore feature parity expectation | ⚠️ | Some legacy patterns differ or are dropped |
| Row-level security on classic no-isolation clusters | ⚠️ | Fine-grained needs shared/serverless UC compute |
| Transactional low-latency lookups | ⚠️ | Governance path adds overhead |

## 10. Architecture Alternatives
| Alternative | Type | Trade-off vs this feature |
|---|---|---|
| Legacy Hive Metastore | In-Databricks (legacy) | Workspace-scoped, no lineage/ABAC/AI governance. |
| Cloud-native IAM only (S3/ADLS policies) | External | Coarse, storage-level; no table/row/column control. |
| AWS Glue Data Catalog / Lake Formation | External | AWS-centric; weaker AI-asset and lineage coverage. |
| Apache Ranger / Atlas | External OSS | Self-managed; integration and lineage effort high. |
| Snowflake Horizon / other warehouse governance | External | Strong in-platform, but tied to that engine. |
| Collibra / Alation | External catalog | Discovery/stewardship strong; not the enforcement point. |

## 11. Strengths
- Single account-level model for grants across all workspaces.
- Three-level namespace (`catalog.schema.object`) organizes data + AI assets consistently.
- Automatic, runtime column-level lineage — no manual instrumentation.
- Row filters, column masks, and ABAC via governed tags.
- Governs beyond tables: volumes, models, functions, agents, vector indexes.
- Open by design — built on Delta and open APIs (Unity Catalog OSS).
- Delta Sharing and Clean Rooms extend governance across organizations.
- Rich audit via system tables for cost, access, and lineage analytics.
- Catalog Explorer + tags enable self-service discovery.
- Enables secure serverless and shared compute (Lakeguard isolation).

## 12. Limitations
- One metastore per region — no single global namespace out of the box.
- Migration from Hive metastore requires deliberate effort and testing.
- Fine-grained controls require UC-compliant (shared/serverless) compute.
- Some legacy cluster/access modes lose features under UC.
- ABAC and newer AI-governance capabilities mature at different rates.
- Cross-cloud governance still spans separate metastores.
- Deep policy design (tags, groups, ABAC) has a real learning curve.
- Lineage covers Databricks-run operations; external writes may be missed.
- External-source governance depth depends on Federation connector coverage.
- Overly broad grants are easy to create without disciplined group design.

## 13. Common Mistakes
- Granting to individual users instead of groups — unmanageable at scale.
- Leaving workloads on the legacy Hive metastore indefinitely.
- Flat catalog design (one catalog, many schemas) with no environment/domain separation.
- Treating `ALL PRIVILEGES` or account-admin as a convenience default.
- Ignoring storage credentials/external location scoping, over-permissioning cloud paths.
- No tagging strategy, so ABAC and discovery never become usable.
- Assuming lineage captures non-Databricks pipelines.
- Not enabling or reviewing system tables/audit logs.
- Designing per-workspace instead of account-first governance.

## 14. Design Recommendations
- ✔ Grant to **groups** (ideally SCIM-synced from your IdP), never individuals.
- ✔ Use catalogs for environment/domain boundaries (`prod`, `dev`, business domain).
- ✔ Adopt a **tag taxonomy early** to power ABAC, discovery, and cost attribution.
- ✔ Scope storage credentials and external locations to least privilege.
- ✔ Standardize on serverless/shared UC compute for fine-grained enforcement.
- ✔ Enable system tables and audit logs from day one.
- ✔ Plan and time-box Hive metastore migration; validate lineage after cutover.
- ✔ Codify grants and objects via Terraform / Asset Bundles for repeatability.

## 18. Cost Considerations
- **Licensing:** included with Databricks; no separate UC license. Verify current pricing.
- **Compute:** governance itself is not billed, but fine-grained enforcement requires serverless/shared compute (which is billed as compute).
- **Storage:** managed tables/volumes consume cloud object storage; predictive optimization maintenance incurs compute.
- **Operational Overhead:** metastore setup, migration, group/tag design, and ongoing stewardship are the main real costs.
- **Training:** upskilling on the namespace model, grants, ABAC, and lineage; budget for a governance owner/CoE.

## 19. Executive Takeaway
- ✔ **Best for:** any organization standardizing data + AI governance across multiple Databricks workspaces.
- ✔ **Avoid when:** you need it as a general application IAM or secret store — that is out of scope.
- ✔ **Biggest benefit:** one consistent permission, lineage, and audit model spanning data and AI.
- ✔ **Biggest limitation:** one metastore per region, so a single global namespace requires design work.
- ✔ **Overall recommendation:** adopt as the mandatory governance foundation; design account-first with group-based grants and a tag taxonomy from day one.
