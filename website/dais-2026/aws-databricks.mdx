# AWS & Databricks at DAIS 2026

> AWS shows up as a Legend Sponsor with Bedrock, AgentCore, and Kiro — but strip the co-marketing and this is a relationship update, not a joint product. The real story is where AWS's agent stack overlaps with Databricks' own.

*Announced at Data + AI Summit 2026 (June 16–17, 2026). Source: [Databricks blog](https://www.databricks.com/blog/aws-and-databricks-data-ai-summit-2026-accelerating-real-world-ai-innovation).*

**Stage:** <StatusBadge stage="Partnership" /> — AWS Legend Sponsor; joint solutions, not a single product.

## What shipped

- **AWS returns as a Legend Sponsor**, positioning **Bedrock, Bedrock AgentCore, and Kiro** alongside Databricks to operationalize AI at scale.
- **Amazon Bedrock** — managed frontier models (**Anthropic Claude, Amazon Nova, Meta Llama**) with built-in **Guardrails**; paired with Databricks they reason over governed, high-quality data.
- **Bedrock AgentCore** — an enterprise **agent runtime** (memory, identity, code-interpreter/browser tools, observability) that queries Unity Catalog-governed data via an agent connection to Databricks.
- **Kiro** — AWS's **agentic IDE** for spec-driven development, turning intent into working code for apps on Databricks and AWS.
- **Named customers on Databricks + AWS:** Mastercard, Talkdesk, nCino, Addepar, Workday — with a Mastercard case study on multi-engine catalog federation.
- **AWS Marketplace offer:** try Databricks with **$400 in usage credits via a 14-day free trial**.

## Where it fits

- **The layer beneath the lakehouse** — most Databricks deployments already run on AWS (S3, compute, networking), so this is alignment at the infrastructure and model layer, not the platform. See [the overview](/dais-2026).
- **Bedrock as an external model provider** — reach Claude, Nova, Llama, and others from Databricks workloads via a managed API, with Guardrails for regulated use.
- **AgentCore for agent hosting** — an option when you want the runtime, memory, and identity managed by AWS while still querying Unity Catalog-governed data.
- **Kiro for the dev loop** — spec-driven agentic coding for teams already living in AWS tooling.
- **Marketplace procurement** — buying Databricks through Marketplace simplifies contracting for AWS-committed enterprises and burns down committed spend.
- **Reference logos** — Mastercard, Talkdesk, nCino, Addepar, Workday signal regulated, large-scale adoption if you need proof for a review board.

## My take

**Verdict: Watch, and pilot only the pieces you already own.** Nothing here forces a decision this quarter. The substance — Databricks runs great on AWS — was already true; Bedrock, AgentCore, and Kiro are AWS trying to win the agent and IDE layers sitting on top. It's a sponsorship narrative dressed as a product story, and mostly ecosystem alignment plus a promotional Marketplace trial.

The honest tension is coopetition. Bedrock and AgentCore **overlap directly with Databricks' own agent stack** (Agent Bricks, Mosaic AI, model serving), so you're choosing whose runtime and models to bet on, not stacking complements. Run both and you get two governance planes — Bedrock Guardrails on one side, Unity Catalog lineage and policy on the other — and reconciling them is on you. Pick one as the source of truth deliberately, or don't split the stack at all.

Practically: pilot **Bedrock** only when Databricks' own serving or partner models don't cover a model you specifically need. Keep your **agent runtime** where your governance and data already live — usually Databricks + Unity Catalog — rather than standing up AgentCore in parallel. **Kiro** is early against Cursor, Copilot, and Claude Code; trial it in a low-stakes repo if you're already an AWS-tooling shop, not before. And weigh the incentives behind the $400 credits — AWS wants your Databricks consumption, which is fine, just not a reason to deepen coupling.

**Bottom line:** If you're on AWS, keep running Databricks there and reach for Bedrock tactically for models you can't get otherwise — but don't split your agent stack for a sponsor's slide.
