# AWS & Databricks at DAIS 2026

> Legend Sponsor AWS shows up with Bedrock, AgentCore, and Kiro to operationalize AI at scale on Databricks — heavier on partnership and co-marketing than on new joint product.

*Announced at Data + AI Summit 2026 (June 16–17, 2026). Source: [Databricks blog](https://www.databricks.com/blog/aws-and-databricks-data-ai-summit-2026-accelerating-real-world-ai-innovation).*

**Stage:** <StatusBadge stage="Partnership" /> — AWS Legend Sponsor; joint solutions, not a single product.

## Overview

- **AWS returns as a Legend Sponsor** and positions **Bedrock, Bedrock AgentCore, and Kiro** alongside Databricks to operationalize AI at scale.
- **Amazon Bedrock** — managed frontier models (**Anthropic Claude, Amazon Nova, Meta Llama**) with built-in **Guardrails**; paired with Databricks they reason over governed, high-quality data.
- **Bedrock AgentCore** — the enterprise **agent runtime** (memory, identity, code-interpreter/browser tools, observability) that securely queries Unity Catalog-governed data via an agent connection to Databricks.
- **Kiro** — AWS's **agentic IDE** for spec-driven development, turning intent into working code for apps that run on Databricks and AWS.
- Named customers running Databricks on AWS: **Mastercard, Talkdesk, nCino, Addepar, Workday** — with a Mastercard case study on multi-engine catalog federation.
- **AWS Marketplace offer**: try Databricks on AWS Marketplace with **$400 in usage credits via a 14-day free trial**.

## Where it fits

- **Infrastructure + model layer** — most Databricks deployments already run on AWS (S3, compute, networking), so this is alignment at the layer beneath the lakehouse. See [the overview](/dais-2026).
- **Bedrock as an external model provider** — reach Anthropic Claude, Amazon Nova, Meta Llama, and other models from Databricks workloads via a managed API, with Guardrails for regulated use.
- **AgentCore for agent hosting** — an option when you want agent runtime, memory, and identity managed by AWS while querying Unity Catalog-governed data.
- **Kiro for the dev workflow** — spec-driven agentic coding for teams building apps on Databricks and AWS.
- **Marketplace procurement** — buying Databricks through AWS Marketplace simplifies contracting for AWS-committed enterprises.
- **Enterprise proof** — the named logos (Mastercard, Talkdesk, nCino, Addepar, Workday) signal regulated, large-scale adoption if you need reference customers.

## Where it fails

- **Partnership, not product** — this reads as co-marketing and ecosystem alignment more than a new joint capability; little here is a Databricks-native feature.
- **Coopetition is real** — Bedrock and AgentCore **overlap with Databricks' own agent stack** (Agent Bricks, Mosaic AI, model serving). You're choosing whose runtime and models to bet on, not stacking complements.
- **Two governance planes** — routing through Bedrock/AgentCore splits observability, guardrails, and lineage between AWS and Unity Catalog; unifying them is on you.
- **Kiro maturity** — an agentic IDE competing with established players (Cursor, Copilot, Claude Code); treat as early for production teams.
- **Lock-in cuts both ways** — leaning on Bedrock models + AgentCore + Marketplace credits deepens AWS coupling and can complicate future multi-cloud or Databricks-native moves.
- **Guardrails overlap** — Bedrock Guardrails vs. Databricks/Unity governance means duplicated policy config unless you deliberately pick one as the source of truth.

## Response

- **Verdict: Watch (pilot the pieces you already own).** The relationship matters if you're an AWS shop, but nothing here forces a new decision this quarter.
- **Pilot Bedrock** as an external model source **only if** Databricks' own model serving or partner models don't cover a model you specifically need (e.g., a Bedrock-exclusive).
- **Watch AgentCore vs. Agent Bricks** — don't run two agent runtimes; pick the one aligned with where your governance and data already live (usually Databricks + Unity Catalog).
- **Try Kiro** only if you're an AWS-tooling shop wanting to trial it in a low-stakes repo; it's early against stronger agentic IDEs.
- **Market context** — mirrors Databricks' multicloud posture (Azure OpenAI, Google) and its own agent push; AWS wants Databricks workloads and consumption, so weigh incentives behind the credits.

## Personal impression

- This is a sponsorship narrative dressed as a product story. The substance — Databricks runs great on AWS — was already true; Bedrock/AgentCore/Kiro are AWS trying to win the agent and IDE layers on top.
- The honest tension is coopetition: AWS's agent runtime and Databricks' agent stack want the same job, and picking both means two governance planes to reconcile.
- **Hype level: Low–Medium.** Real partnership, real customers, but mostly ecosystem alignment plus a promotional Marketplace trial.
- **Bottom line:** If you're on AWS, keep running Databricks there and use Bedrock tactically for models you can't get otherwise — but don't split your agent stack for a sponsor's slide.
