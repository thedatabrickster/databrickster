# Omnigent

> A harness over harnesses: one interface to swap, sandbox, and govern Claude Code, Codex, Pi, and custom agents — with policy and live collaboration built in. The idea is right; the maturity isn't there yet.

*Announced at Data + AI Summit 2026 (June 16–17, 2026). Source: [Databricks blog](https://www.databricks.com/blog/introducing-omnigent-meta-harness-combine-control-and-share-your-agents).*

**Stage:** <StatusBadge stage="Alpha" /> — open source (Apache-2.0).

## What it is

- **An Apache-2.0 meta-harness** that sits *above* your coding agents — one common interface to combine, control, and share Claude Code, Codex, Pi, and custom agents. Repo: [omnigent-ai/omnigent](https://github.com/omnigent-ai/omnigent).
- **Alpha, open source, out now** — an early release from Databricks engineers including Matei Zaharia, Kasey Uhlenhuth, and Corey Zumar. Real code, not a slide.
- **One-line agent swaps** — move between Claude Code, Codex, Pi, and your own agents, and route different LLMs to planning, search, and code generation.
- **Stateful policies** — track agent actions and enforce guardrails like cost budgets and permissions (e.g. pause and ask before continuing past every $100 spent).
- **Live session sharing** — invite others to watch a session, comment on files in its workspace, or send commands, in real time.
- **Two parts** — a **Runner** wraps any agent in a sandboxed session behind a uniform API; a **Server** layers on policy and sharing, exposing every session over terminal, app, and web.

## Where it fits

- **Orchestration/control layer for dev tooling** — it sits between engineers and the agents they run, not on the data plane. See [the overview](/dais-2026).
- The target user is a team running **more than one coding agent** that wants a single control surface instead of per-tool config and per-tool governance.
- **Planning-vs-codegen split** is the sharpest idea here: a strong reasoning model to plan, a cheaper/faster one to generate. That maps to real cost and latency wins.
- **Cost budgets plus sandboxed sessions** hit the actual enterprise blocker — letting agents run with guardrails an auditor will sign off on.
- **Shared live sessions** fit pairing, review, and incident debugging where two engineers want to steer one agent together.

## Where it falls short

- **Alpha maturity.** Thin docs, rough edges, no production track record. This is pilot material, not a rollout — budget for breakage.
- **Another abstraction layer.** A harness over harnesses adds indirection, its own bugs, and constant version-skew against tools that ship weekly (Claude Code, Codex). You inherit everyone's release cadence.
- **Lowest-common-denominator risk.** A single common interface tends to surface only what all agents share, blunting the features that make each one worth using.
- **Adoption risk vs. one harness directly.** Most teams standardize on a single agent, where direct use is simpler and loses almost nothing. The multi-agent premise only pays off at genuine tool diversity — which most orgs don't have.
- **The one-line swap rarely holds clean** — tool schemas, prompt behavior, and permission models differ per agent, so portability leaks in practice.

## My take

The architecture is genuinely good. Sandboxed sessions, cost pauses, and the planning/codegen model split are real gaps in today's single-harness workflows, and I'd happily use those parts tomorrow. That it ships as an actual Apache-2.0 repo from a credible team — Zaharia, Uhlenhuth, Zumar — is not nothing.

But a meta-harness only earns its keep when you're already juggling multiple agents and feeling the governance and cost pain. Most teams aren't there. For them, Omnigent is one more layer to babysit against tools that move faster than it can — and alpha maturity means you're debugging the abstraction, not the work.

So: **watch, don't adopt.** Pilot only if you run two-plus coding agents today and the cost/policy pain is real — put the sandbox and budget claims on one team and see if they hold. If you've standardized on a single agent, stay direct.

**Bottom line:** The right idea from the right people, a full generation too early — bookmark the repo, run a pilot only if multi-agent governance already hurts.
