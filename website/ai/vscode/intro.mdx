# AI Engineering in VS Code & Claude Code

:::note Coming soon
This topic is a **placeholder** — it's on the roadmap. We'll build it out for both
**VS Code** and **Claude Code**, the same way as [Databricks AI](/docs/intro).
Here's the plan.
:::

> Notebooks are great for exploring. But real AI projects are *software* — with
> repos, tests, reviews, and CI. This topic is about being a fast, professional AI
> engineer in a real editor: **VS Code**, supercharged by **Claude Code**.

## What this topic will cover

This track is tool-focused and useful whether you build on Databricks or anywhere
else. Planned lessons:

- **Why the IDE, not just notebooks** — when to move from notebooks to a versioned, testable codebase, and how the two work together.
- **Set up VS Code for Databricks** — the Databricks extension, Databricks Connect, running code on remote compute, and Asset Bundles from the editor.
- **Claude Code for AI engineers** — using an agentic coding assistant to scaffold agents, write tests, refactor, and debug; what it's great at and where to keep a human in the loop.
- **A repo-first agent workflow** — structure a Databricks AI project as code (Asset Bundles), with local iteration and CI/CD.
- **Testing & debugging agents locally** — tracing, evaluation, and reproducing failures outside a notebook.
- **MCP in your editor** — connecting Claude Code / VS Code to Databricks tools (Unity Catalog functions, Vector Search, Genie) via the Model Context Protocol.
- **General AI engineering** — the same workflow applied off Databricks (any model provider, any stack).

## How it connects to the rest of the site

The [Databricks AI](/docs/intro) track teaches *what* to build; this track teaches
*how to build it fast and professionally* in a real development environment. It also
pairs naturally with [Part 11 · Beyond Databricks](/docs/beyond-databricks/concepts-are-portable),
since a good editor workflow is portable across any AI stack.

*Two editors, one goal: ship AI faster. On the roadmap.*
